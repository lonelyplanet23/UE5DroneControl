## 一、后端整体架构

### 1.1 模块划分

```
后端服务
│
├── 注册管理模块        无人机的增删改查与持久化，ID↔编号映射
├── 队列管理模块        每架无人机的指令队列、心跳维持、连接状态机
├── 执行引擎            集结流程、侦察/巡逻/攻击模式逻辑、阵列任务调度、暂停/恢复、实时避障
├── 通讯模块
│   ├── 对 UE 端       HTTP REST Server + WebSocket Server
│   └── 对无人机端     UDP 收发（控制包 + 遥测包）
└── 坐标转换模块        NED↔UE 偏移转换、四元数处理、GPS 锚点管理
```

### 1.2 对外接口总览

**HTTP REST（供 UE5 调用）**

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/drones` | GET | 获取所有注册无人机列表+状态 |
| `/api/drones` | POST | 注册新无人机 |
| `/api/drones/{id}` | PUT | 更新无人机信息 |
| `/api/drones/{id}` | DELETE | 删除无人机 |
| `/api/drones/{id}/anchor` | GET | 获取无人机 GPS 锚点坐标 |
| `/api/arrays` | POST | 下发阵列任务 |
| `/api/arrays/{id}/stop` | POST | 停止阵列任务 |
| `/api/debug/drone/{id}/state` | GET | **[测试]** 查询无人机完整内部状态 |
| `/api/debug/drone/{id}/queue` | GET | **[测试]** 查询无人机当前指令队列 |
| `/api/debug/drone/{id}/inject` | POST | **[测试]** 注入模拟遥测数据 |
| `/api/debug/heartbeat/{id}` | GET | **[测试]** 查询心跳发送统计 |
| `/api/debug/cmd/{id}/move` | POST | **[测试]** 模拟 UE5 发送单点移动指令 |
| `/api/debug/cmd/{id}/pause` | POST | **[测试]** 模拟 UE5 发送暂停指令 |
| `/api/debug/cmd/{id}/resume` | POST | **[测试]** 模拟 UE5 发送恢复指令 |
| `/api/debug/cmd/{id}/array` | POST | **[测试]** 模拟 UE5 下发阵列任务 |
| `/api/debug/cmd/{id}/target` | POST | **[测试]** 注入目标识别事件（巡逻模式） |
| `/api/debug/cmd/batch/array` | POST | **[测试]** 多机并发阵列任务下发 |
| `/api/debug/arrays/{id}/state` | GET | **[测试]** 查询阵列任务当前状态（ASSEMBLING/EXECUTING/DONE） |

**WebSocket（供 UE5 订阅）**

| 方向 | 消息类型 | 说明 |
|------|----------|------|
| UE → 后端 | `move` | 目标点指令（UE 偏移，厘米） |
| UE → 后端 | `pause` | 暂停指定无人机 |
| UE → 后端 | `resume` | 恢复指定无人机 |
| 后端 → UE | `telemetry` | 遥测推送（位置/姿态/电量/速度） |
| 后端 → UE | `event` | 状态事件（power_on / reconnect / lost_connection） |
| 后端 → UE | `assembling` | 集结进度（ready_count / total_count） |
| 后端 → UE | `assembly_complete` | 集结完成，开始执行阵列 |
| 后端 → UE | `assembly_timeout` | 集结超时告警 |
| 后端 → UE | `alert` | 告警（低电量 / 断联 / 集结超时） |

**UDP（与 PX4 无人机通信）**

| 方向 | 格式 | 频率 |
|------|------|------|
| 后端 → 无人机 | 24 字节控制包（小端） | ≥2Hz 心跳，派发时高频 |
| 无人机 → 后端 | YAML 遥测包 | 10Hz |

### 1.3 数据流

```
PX4 无人机
  │  UDP YAML 遥测（10Hz）
  ↓
后端：解析 → 坐标转换（NED→UE 偏移）→ WebSocket 推送给 UE5
后端：记录 GPS 锚点 → power_on 事件推送给 UE5

UE5
  │  WebSocket move 指令（UE 偏移，厘米）
  ↓
后端：坐标转换（UE 偏移→NED）→ 入队 → UDP 24字节 → PX4 无人机
```

## 二、需求拆分

### 2.1 注册管理模块

| 需求名称 | 需求详细描述 | 完成人员 | 测试人员 | 测试方法 |
|----------|-------------|----------|----------|----------|
| 无人机注册 | 接收 UE5 POST `/api/drones` 请求，持久化无人机信息（名称、型号、IP:Port、视频流地址），自动分配唯一 ID，返回分配的 ID | 关皓元 | | 调用 POST `/api/drones`，检查返回 ID 唯一且递增；重复名称应返回 4xx 错误 |
| 无人机查询 | 接收 GET `/api/drones` 请求，返回所有已注册无人机列表，每条记录包含 ID、名称、型号、IP:Port、当前连接状态 | 关皓元 | | 注册若干架后调用 GET，验证返回数量和字段完整性 |
| 无人机更新 | 接收 PUT `/api/drones/{id}` 请求，允许修改除 ID 外的所有字段，持久化更新 | 关皓元 | | 修改名称后重新 GET，验证字段已更新 |
| 无人机删除 | 接收 DELETE `/api/drones/{id}` 请求，从持久化存储中移除，同时清理该无人机的队列和状态 | 关皓元 | | 删除后 GET 列表验证已移除；删除不存在 ID 应返回 404 |
| ID↔编号映射 | 维护无人机 ID 与通讯端口编号（1~6）的映射关系，编号由用户在注册时手动指定，固定对应 UDP 端口；已被占用的编号不可重复分配，注册请求中若编号已占用应返回 4xx 错误；删除无人机时释放对应编号 | 关皓元 | | 注册时指定编号 1，再次注册指定编号 1 应返回错误；删除后重新注册编号 1 应成功；调用 `/api/debug/drone/{id}/state` 验证编号字段与注册时一致 |
| 持久化存储 | 注册数据写入本地文件或数据库，服务重启后数据不丢失 | 关皓元 | | 注册后重启服务，GET 列表验证数据仍存在 |

### 2.2 队列管理模块

| 需求名称 | 需求详细描述 | 完成人员 | 测试人员 | 测试方法 |
|----------|-------------|----------|----------|----------|
| 独立指令队列 | 每架无人机维护独立的 FIFO 指令队列，队列满时丢弃最旧指令（或按策略处理），多机互不干扰 | 陈锦垚 | | 同时向 2 架无人机发送不同指令，调用 `/api/debug/drone/{id}/queue` 验证各自队列内容独立 |
| 心跳维持 | 对每架在线无人机，以 ≥2Hz 频率持续发送 24 字节 UDP 保活包（Mode=0 悬停），防止 PX4 退出 Offboard 模式 | 陈锦垚 | | 调用 `/api/debug/heartbeat/{id}` 验证发送频率 ≥2Hz；断开 UE5 连接后验证心跳仍持续 |
| 连接状态机 | 维护每架无人机的连接状态：离线→连接中→在线→失联，状态变更时通过 WebSocket 推送 `event` 消息给 UE5 | 陈锦垚 | | 模拟无人机上线/断线，调用 GET `/api/drones` 验证状态字段变化；同时验证 WebSocket 收到对应 event |
| 超时失联检测 | 若超过 10 秒未收到无人机遥测包，将状态置为"失联"，推送 `lost_connection` 事件，停止心跳发送 | 陈锦垚 | | 停止模拟遥测注入（`/api/debug/drone/{id}/inject` 停止），等待 10s 后验证状态变为失联且 WebSocket 收到事件 |
| 重连处理 | 失联后重新收到遥测包时，重新进入"在线"状态，重新建立心跳，推送 `reconnect` 事件并携带新 GPS 锚点 | 陈锦垚 | | 失联后重新注入遥测数据，验证状态恢复、心跳恢复、WebSocket 收到 reconnect 事件 |

### 2.3 执行引擎

| 需求名称 | 需求详细描述 | 完成人员 | 测试人员 | 测试方法 |
|----------|-------------|----------|----------|----------|
| 集结流程 | 接收 POST `/api/arrays` 后，进入 ASSEMBLING 状态；根据阵列槽位映射计算每架无人机的初始目标点（NED），向各机发送集结指令；实时监控各机位置，到达初始槽位（误差 < 阈值）后计数 +1，并推送 `assembling` 进度事件；全部到位后推送 `assembly_complete` 并进入 EXECUTING 状态；超过 60s（可配置）未全部到位则推送 `assembly_timeout` 告警并中止 | | | 调用 `/api/debug/cmd/batch/array` 下发 3 机阵列任务，通过 `/api/debug/drone/{id}/inject` 模拟各机逐步到达初始槽位，验证 WebSocket 依次收到 assembling（进度递增）→ assembly_complete；模拟其中一机超时未到位，验证收到 assembly_timeout |
| 实时避障 | 在集结阶段和阵列执行阶段，持续监控所有在线无人机的当前 NED 位置；若预测两机在未来 T 秒内距离小于安全阈值（默认 3m，可配置），则对其中一机（优先级低者）临时调整目标点以绕行，绕行完成后恢复原目标点；避障期间不中断心跳 | | | 通过 `/api/debug/drone/{id}/inject` 模拟两机位置逐渐接近至 3m 以内，验证后端对其中一机发出调整指令（抓 UDP 包）；两机分开后验证恢复原目标点 |
| 侦察模式 | 接收阵列任务后，按航点序列依次向无人机发送目标点指令；判断无人机到达当前航点（位置误差 < 阈值）后，自动发送下一个航点；支持循环模式 | | | 调用 `/api/debug/cmd/{id}/array` 下发侦察模式阵列，再通过 `/api/debug/drone/{id}/inject` 模拟无人机位置逐步逼近各航点，验证后端依次推进；循环模式验证到达末航点后回到第一个 |
| 巡逻模式 | 在侦察模式基础上，支持接收"目标识别"事件（由 UE5 通过 WebSocket 发送），收到事件后中断当前航点序列，飞向目标点；目标点到达后停止 | | | 调用 `/api/debug/cmd/{id}/array` 下发巡逻模式阵列，飞行途中调用 `/api/debug/cmd/{id}/target` 注入目标识别事件，验证指令切换到目标点；到达后验证停止 |
| 攻击模式 | 接收阵列任务后，按航点序列依次飞行（与侦察模式相同），到达最后一个航点（攻击目标点）后悬停，不循环；与侦察模式的区别在于语义上最后一个航点为攻击目标，且到达后不继续 | | | 通过 `/api/debug/cmd/{id}/array` 下发含 3 个航点的攻击模式阵列，注入遥测模拟逐步到达，验证依次经过中间航点后到达最终目标点并悬停，不循环 |
| 阵列任务调度 | 接收 POST `/api/arrays` 请求，解析槽位映射，为每架无人机启动独立执行协程/线程，各自按模式执行，互不阻塞 | | | 调用 `/api/debug/cmd/batch/array` 同时下发 3 架无人机的阵列任务，验证 3 架同时开始执行，各自队列独立推进 |
| 暂停/恢复 | 接收 WebSocket `pause` 指令，立即向对应无人机发送悬停包（Mode=0），暂停队列推进；接收 `resume` 后恢复 | | | 调用 `/api/debug/cmd/{id}/pause` 发送暂停，验证 `/api/debug/drone/{id}/queue` 队列停止推进；调用 `/api/debug/cmd/{id}/resume` 验证恢复 |
| 到达判断阈值 | 判断无人机是否到达目标点的位置误差阈值可配置（默认 1 米），通过配置文件或启动参数设置 | | | 修改阈值为 0.5m 和 5m，验证航点推进时机变化符合预期 |

## 三、配置文件结构

后端无 GUI，所有运行参数通过 `config.yaml` 配置文件管理，启动时读取，修改后重启生效。

```yaml
server:
  http_port: 8080          # HTTP REST 服务端口
  ws_port: 8081            # WebSocket 服务端口
  debug: true              # 是否启用测试接口（/api/debug/...），生产环境设为 false

drone:
  max_count: 6             # 最大注册无人机数量
  heartbeat_hz: 2          # 心跳发送频率（Hz），不得低于 2
  lost_timeout_sec: 10     # 超时失联阈值（秒）
  arrival_threshold_m: 1.0 # 到达航点的位置误差阈值（米）
  assembly_timeout_sec: 60 # 集结超时阈值（秒）
  avoidance_radius_m: 3.0  # 实时避障安全距离阈值（米）
  avoidance_lookahead_sec: 2.0 # 避障预测时间窗口（秒）
  low_battery_threshold: 20 # 低电量告警阈值（%）

# 编号与 UDP 端口的固定映射（编号由用户注册时手动指定）
# 编号 1~6 分别对应以下收发端口，不可动态修改
port_map:
  1:
    send_port: 8889        # 后端向该编号无人机发送控制包的目标端口
    recv_port: 8888        # 后端监听该编号无人机遥测包的本地端口
  2:
    send_port: 8891
    recv_port: 8890
  3:
    send_port: 8893
    recv_port: 8892
  4:
    send_port: 8895
    recv_port: 8894
  5:
    send_port: 8897
    recv_port: 8896
  6:
    send_port: 8899
    recv_port: 8898

storage:
  path: "./data/drones.json"  # 注册数据持久化文件路径

log:
  level: "info"            # 日志级别：debug / info / warn / error
  file: "./logs/backend.log" # 日志文件路径，留空则只输出到终端
```

> 编号与端口的映射关系（`port_map`）在部署前由运维/开发人员根据实际网络配置填写，运行期间不变。用户注册无人机时选择编号 1~6，后端据此查表确定通信端口。

## 四、测试接口说明

以下接口仅用于开发测试阶段，生产环境可通过配置关闭（`debug: false`）：

**状态查询类**

| 接口 | 方法 | 说明 | 请求体 |
|------|------|------|--------|
| `/api/debug/drone/{id}/state` | GET | 返回无人机完整内部状态（连接状态、队列长度、锚点、最近遥测时间戳等） | — |
| `/api/debug/drone/{id}/queue` | GET | 返回当前指令队列内容（NED 坐标列表） | — |
| `/api/debug/heartbeat/{id}` | GET | 返回心跳发送统计（频率、最近发送时间、累计发送次数） | — |

**数据注入类**（模拟无人机，替代真实硬件）

| 接口 | 方法 | 说明 | 请求体示例 |
|------|------|------|-----------|
| `/api/debug/drone/{id}/inject` | POST | 注入一帧模拟遥测数据，触发后端完整处理流程 | `{"position":[1.0,2.0,-10.0],"q":[1,0,0,0],"velocity":[0,0,0],"battery":85,"gps_lat":39.9,"gps_lon":116.3,"gps_alt":50.0}` |

**指令发送类**（模拟 UE5，替代前端）

| 接口 | 方法 | 说明 | 请求体示例 |
|------|------|------|-----------|
| `/api/debug/cmd/{id}/move` | POST | 向指定无人机发送单个目标点指令（UE 偏移，厘米） | `{"x":1000,"y":2000,"z":500}` |
| `/api/debug/cmd/{id}/pause` | POST | 向指定无人机发送暂停指令 | — |
| `/api/debug/cmd/{id}/resume` | POST | 向指定无人机发送恢复指令 | — |
| `/api/debug/cmd/{id}/array` | POST | 向指定无人机下发阵列任务（含模式和航点序列） | `{"mode":"recon","loop":false,"waypoints":[{"x":1000,"y":0,"z":500},{"x":2000,"y":1000,"z":500}]}` |
| `/api/debug/cmd/{id}/target` | POST | 向指定无人机注入目标识别事件（用于测试巡逻模式脱离序列） | `{"x":3000,"y":3000,"z":500}` |
| `/api/debug/cmd/batch/array` | POST | 同时向多架无人机下发阵列任务（测试多机并发调度） | `[{"drone_id":"d1","mode":"recon","waypoints":[...]},{"drone_id":"d2","mode":"attack","waypoints":[...]}]` |

> 指令发送类接口使后端执行引擎可在 UE5 尚未开发完成时独立测试，所有执行模式逻辑均可通过这些接口覆盖。

## 四、关键约束与注意事项

| 约束 | 说明 |
|------|------|
| 心跳频率 | 必须 ≥2Hz，否则 PX4 自动退出 Offboard 模式，无人机悬停 |
| 最大无人机数 | 6 架，超出注册请求应返回错误 |
| 坐标单位 | UE5 端传入/传出均为厘米，无人机端均为米，转换系数 ×100 / ×0.01 |
| move 指令坐标语义 | UE5 发来的 x/y/z 是相对锚点的 UE 偏移（厘米），不是绝对世界坐标，后端直接做单位转换即可 |
| 遥测推送坐标语义 | 推送给 UE5 的 x/y/z 是相对锚点的 UE 偏移（厘米），UE5 端叠加锚点世界坐标得到最终位置 |
| 告警去重 | 同一无人机同类告警触发后，状态未恢复前不重复推送 |
| 语言选型 | Go / Rust / C++，具体由后端团队确定，需支持高并发 UDP 收发和 WebSocket 推送 |

### 开发周期安排

**第一周：** 基础架构搭建 + 学习期（后端框架选型、配置文件、注册接口基础）

**第三周：** 实时控制 + 状态管理（连接状态机、move/pause/resume 指令、遥测推送）
**第四周：** 阵列执行 + 集结流程（集结状态机、三种执行模式、多机并发调度）
**第五周：** 完善功能 + 集成测试（避障优化、告警推送、全流程联调）
**第六周：** 冗余周（Bug 修复 + 优化 + 文档完善）
