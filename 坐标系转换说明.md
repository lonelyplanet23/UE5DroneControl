# 坐标系转换说明

> 版本：v0.1
> 日期：2026-04-13
> 面向：UE5 工程师、后端工程师、测试人员

---

## 一、坐标系定义速查

| 坐标系 | 使用方 | 轴向定义 | 单位 | 原点 |
|--------|--------|----------|------|------|
| **NED**（北东下） | PX4 无人机 | X=North（北），Y=East（东），Z=Down（下） | 米 | 无人机上电时的位置 |
| **UE5 世界坐标** | UE5 引擎 | X=Forward（前），Y=Right（右），Z=Up（上） | 厘米 | UE5 场景原点 |
| **ECEF** | Cesium 内部 | 地心地固坐标系 | 米 | 地球质心 |
| **GPS**（WGS84） | 地理定位 | lat/lon/alt | 度/米 | 地球椭球面 |

> **重要**：UE5 使用**左手坐标系**，NED 使用**右手坐标系**，转换时需注意手性差异对 Yaw 方向的影响。

## 二、链路 A：无人机 → 后端 → UE5（遥测数据）

### 2.1 数据流概览

```
PX4 无人机
  │  ROS2 话题：/px4_N/fmu/out/vehicle_odometry
  │  坐标系：NED，单位：米，原点：上电位置
  ↓
后端服务
  │  接收 UDP YAML 遥测包（10Hz）
  │  执行 NED → UE 偏移量转换
  │  执行四元数重排与 Yaw 翻转
  ↓
UE5 前端
     接收 WebSocket 遥测推送
     驱动镜像机位置和姿态
```

### 2.2 无人机侧数据格式（YAML 遥测包）

PX4 通过 ROS2 话题发布 odometry 数据，后端接收后解析以下字段：

| 字段 | 类型 | 坐标系 | 说明 |
|------|------|--------|------|
| `timestamp` | uint64 | — | 时间戳（微秒） |
| `position` | float[3] | NED，米 | [North, East, Down]，相对上电原点 |
| `q` | float[4] | FRD 机体系 | YAML 顺序为 **[w, x, y, z]** |
| `velocity` | float[3] | NED，m/s | [vNorth, vEast, vDown] |
| `angular_velocity` | float[3] | — | 角速度（rad/s） |

### 2.3 后端坐标转换（NED → UE 偏移量）

后端完成以下转换，结果以 UE5 世界坐标偏移量（厘米）推送给 UE：

**位置转换**：

```
UE_offset_X = NED_North × 100    // 北 → UE X，单位：厘米
UE_offset_Y = NED_East  × 100    // 东 → UE Y，单位：厘米
UE_offset_Z = NED_Down  × (-100) // 下 → UE Z（取反），单位：厘米
```

**四元数转换**（NED/FRD → UE5 左手系）：

```
// 输入：YAML 中 q = [w, x, y, z]
// 重排为标准四元数：FQuat(x, y, z, w)
// 翻转 Z 分量以适配 UE5 左手系：FQuat(x, y, -z, w)
// Yaw 角同步取反：Yaw_UE = -Yaw_NED
```

### 2.4 后端推送给 UE 的 WebSocket 消息格式

```json
{
  "type": "telemetry",
  "drone_id": "d1",
  "x": 10000,       // UE 世界坐标偏移 X，厘米
  "y": 20000,       // UE 世界坐标偏移 Y，厘米
  "z": 5000,        // UE 世界坐标偏移 Z，厘米
  "yaw": 45.0,      // UE5 Yaw 角，度（已翻转）
  "pitch": 0.0,     // 俯仰角，度
  "roll": 0.0,      // 横滚角，度
  "speed": 3.5,     // 速度，m/s
  "battery": 85     // 电量，%
}
```

> 注意：`x/y/z` 是相对无人机上电锚点的**偏移量**，不是 UE5 世界绝对坐标。UE5 端需将其叠加到锚点世界坐标上，才能得到镜像机的最终世界坐标。

### 2.5 UE5 端最终定位计算

UE5 收到遥测数据后，镜像机世界坐标计算方式：

```
MirrorWorldLocation = AnchorWorldLocation + UE_offset

其中：
  AnchorWorldLocation = Cesium.TransformGPSToUnreal(anchor_lat, anchor_lon, anchor_alt)
  UE_offset = (x, y, z)  // 后端推送的偏移量（厘米）
```

**锚点建立流程**：
1. 后端收到无人机首包遥测时，记录当前 GPS 坐标作为锚点
2. 后端通过 WebSocket 推送 `power_on` 事件，携带 `gps_lat/gps_lon/gps_alt`
3. UE5 收到事件后，调用 Cesium API 将 GPS 转换为 UE5 世界坐标，存为 `AnchorWorldLocation`
4. 后续每帧：`MirrorWorldLocation = AnchorWorldLocation + UE_offset`

## 三、链路 B：UE5 → 后端 → 无人机（控制指令）

### 3.1 数据流概览

```
UE5 前端
  │  用户点击地图，生成目标点（UE5 世界坐标，厘米）
  │  减去本地存储的 AnchorWorldLocation，得到相对偏移
  │  通过 WebSocket 发送 move 指令（相对偏移，厘米）
  ↓
后端服务
  │  接收 WebSocket 控制指令（已是相对偏移，无需再减锚点）
  │  执行 UE 偏移 → NED 转换
  │  打包为 24 字节 UDP 控制包
  ↓
PX4 无人机
     接收 UDP 控制包，执行飞行
```

### 3.2 UE5 端预处理（发送前）

UE5 在发送指令前，先将目标点从绝对世界坐标转换为相对锚点的偏移：

```
UE_offset_X = UE_target_X - AnchorWorldLocation_X
UE_offset_Y = UE_target_Y - AnchorWorldLocation_Y
UE_offset_Z = UE_target_Z - AnchorWorldLocation_Z
```

> `AnchorWorldLocation` 存储在 UE5 端，由锚定流程（见 2.5）建立。后端不持有此数据。

### 3.3 UE5 发送给后端的 WebSocket 消息格式

```json
{
  "type": "move",
  "drone_id": "d1",
  "x": 1000,    // 相对锚点的 UE 偏移 X，厘米
  "y": 2000,    // 相对锚点的 UE 偏移 Y，厘米
  "z": 500,     // 相对锚点的 UE 偏移 Z，厘米
  "mode": 1     // 0=悬停，1=移动
}
```

> `x/y/z` 是相对无人机上电锚点的 UE 坐标偏移（厘米），与遥测推送中的偏移量坐标系一致。后端收到后直接做单位转换，无需再减锚点。

### 3.4 后端坐标转换（UE 偏移 → NED）

```
NED_North = UE_offset_X × 0.01    // 厘米 → 米
NED_East  = UE_offset_Y × 0.01    // 厘米 → 米
NED_Down  = UE_offset_Z × (-0.01) // 取反，厘米 → 米
```

### 3.5 后端发送给无人机的 UDP 控制包（24 字节）

```c
struct DroneControlPacket {
    double  Timestamp;   // 8 bytes，Unix 时间戳（秒）
    float   X;           // 4 bytes，NED North 偏移（米）
    float   Y;           // 4 bytes，NED East 偏移（米）
    float   Z;           // 4 bytes，NED Down 偏移（米）
    int32   Mode;        // 4 bytes，0=悬停，1=移动
};
// 字节序：小端（Little-Endian）
// Python 解析：struct.unpack('<dfffi', data)
```

> 坐标原点：无人机上电时的 NED 原点（与 odometry 原点一致）。

## 四、GPS 锚点与 Cesium 坐标转换（特殊说明）

### 4.1 为什么需要 GPS 锚点

PX4 的 odometry `position` 字段是相对上电位置的 NED 偏移，**不是地理绝对坐标**。要在 Cesium 地图上正确显示无人机位置，必须知道上电时的 GPS 坐标（锚点），才能将 NED 偏移映射到地球上的真实位置。

### 4.2 GPS → UE5 世界坐标（Cesium API）

此转换**只在 UE5 端执行**，后端不参与：

```
AnchorWorldLocation = CesiumGeoreference.TransformLongitudeLatitudeHeightToUnreal(
    lon,   // 经度（度）
    lat,   // 纬度（度）
    alt    // 高度（米，WGS84 椭球高）
)
// 返回值：UE5 世界坐标（厘米）
```

### 4.3 为什么不用简单线性公式替代 Cesium API

简单线性公式（`UE_X = NED_North × 100`）存在两个问题：

1. **轴向不对齐**：Cesium 默认在 Georeference 原点处 UE5 +X 对应东方向，不是北方向，直接套公式会导致 North/East 接反。
2. **球面误差**：线性公式假设地球是平的，离 Georeference 原点越远误差越大（几公里范围内误差显著）。

**正确做法**：
- 锚点 GPS → UE5 世界坐标：用 Cesium API（UE5 端）
- NED 偏移 → UE5 偏移：用线性公式（后端，近距离范围内可接受）
- 最终位置 = Cesium 锚点 + 线性偏移

## 五、转换公式汇总

### 链路 A（遥测，无人机 → UE5）

| 步骤 | 执行方 | 输入 | 输出 | 公式 |
|------|--------|------|------|------|
| NED → UE 偏移 | 后端 | NED (m) | UE offset (cm) | X×100, Y×100, Z×(-100) |
| 四元数重排 | 后端 | [w,x,y,z] | FQuat | (x, y, -z, w)，Yaw 取反 |
| GPS → UE 世界坐标 | UE5 | lat/lon/alt | UE world (cm) | Cesium API |
| 最终定位 | UE5 | 锚点 + 偏移 | 镜像机位置 | AnchorLoc + UE_offset |

### 链路 B（控制，UE5 → 无人机）

| 步骤 | 执行方 | 输入 | 输出 | 公式 |
|------|--------|------|------|------|
| UE 绝对 → UE 偏移 | **UE5** | UE world (cm) | UE offset (cm) | target - AnchorWorldLocation |
| UE 偏移 → NED | 后端 | UE offset (cm) | NED (m) | X×0.01, Y×0.01, Z×(-0.01) |
| 打包发送 | 后端 | NED (m) | 24字节 UDP | struct '<dfffi' |

## 六、断联重连处理

无人机断联后重新上电，odometry 原点重置（新的上电位置），旧锚点失效。

**重连流程**：
1. 后端检测到重连，重新记录新的 GPS 锚点
2. 后端通过 WebSocket 推送 `reconnect` 事件，携带新的 `gps_lat/gps_lon/gps_alt`
3. UE5 收到事件，重新调用 Cesium API 计算新锚点，更新 `AnchorWorldLocation`
4. 镜像机位置从新锚点重新开始计算

## 七、常见错误排查

| 现象 | 可能原因 | 检查点 |
|------|----------|--------|
| 镜像机位置偏移方向错误 | NED 轴向接反 | 确认 X=North, Y=East, Z=Down |
| 镜像机 Z 轴方向相反 | Z 取反遗漏 | 检查 `NED_Down × (-100)` |
| 镜像机 Yaw 方向相反 | Yaw 翻转遗漏 | 检查 `Yaw_UE = -Yaw_NED` |
| 镜像机位置整体偏移 | 锚点未正确建立 | 检查 `power_on` 事件和 Cesium API 调用 |
| 重连后镜像机位置跳变 | 锚点未更新 | 检查 `reconnect` 事件处理逻辑 |
| 控制指令目标点偏移 | 未在 UE5 端减去锚点 | 检查 UE5 发送前是否执行 target - AnchorWorldLocation |
| 单位错误（米/厘米混淆） | 乘以系数错误 | UE5 内部用厘米，NED 用米，×100 或 ×0.01 |
