# 坐标系转换说明

> 版本：v0.3
> 日期：2026-05-23（补充 Cesium API 使用细节、轴向对齐说明、调试经验）
> 面向：UE5 工程师、后端工程师、测试人员

---

## 一、坐标系定义速查

| 坐标系 | 使用方 | 轴向定义 | 单位 | 原点 |
|--------|--------|----------|------|------|
| **NED**（北东下） | PX4 无人机 | X=North（北），Y=East（东），Z=Down（下） | 米 | 无人机上电时的位置 |
| **UE5 世界坐标** | UE5 引擎 | X=Forward（前），Y=Right（右），Z=Up（上） | 厘米 | UE5 场景原点 |
| **ECEF** | Cesium 内部 | 地心地固坐标系 | 米 | 地球质心 |
| **GPS**（WGS84） | 地理定位 | lat/lon/alt | 度/米 | 地球椭球面 |

> **重要**：UE5 使用**左手坐标系**，NED 使用**右手坐标系**，转换时需注意手性差异对 Yaw 方向的影响。

## 二、链路 A：无人机 → 后端 → UE5（遥测数据）

### 2.1 数据流概览

```
PX4 无人机
  │  ROS2 话题：/px4_N/fmu/out/vehicle_odometry
  │  坐标系：NED，单位：米，原点：上电位置
  ↓
后端服务
  │  接收 UDP YAML 遥测包（10Hz）
  │  执行 NED → UE 偏移量转换
  │  执行四元数重排与 Yaw 翻转
  ↓
UE5 前端
     接收 WebSocket 遥测推送
     驱动镜像机位置和姿态
```

### 2.2 无人机侧数据格式（YAML 遥测包）

PX4 通过 ROS2 话题发布 odometry 数据，后端接收后解析以下字段：

| 字段 | 类型 | 坐标系 | 说明 |
|------|------|--------|------|
| `timestamp` | uint64 | — | 时间戳（微秒） |
| `position` | float[3] | NED，米 | [North, East, Down]，相对上电原点 |
| `q` | float[4] | FRD 机体系 | YAML 顺序为 **[w, x, y, z]** |
| `velocity` | float[3] | NED，m/s | [vNorth, vEast, vDown] |
| `angular_velocity` | float[3] | — | 角速度（rad/s） |

### 2.3 后端坐标转换（NED → UE 偏移量）

后端完成以下转换，结果以 UE5 世界坐标偏移量（厘米）推送给 UE：

**位置转换**：

```
UE_offset_X = NED_North × 100    // 北 → UE X，单位：厘米
UE_offset_Y = NED_East  × 100    // 东 → UE Y，单位：厘米
UE_offset_Z = NED_Down  × (-100) // 下 → UE Z（取反），单位：厘米
```

**四元数转换**（NED/FRD → UE5 左手系）：

```
// 输入：YAML 中 q = [w, x, y, z]
// 重排为标准四元数：FQuat(x, y, z, w)
// 翻转 Z 分量以适配 UE5 左手系：FQuat(x, y, -z, w)
// Yaw 角同步取反：Yaw_UE = -Yaw_NED
```

### 2.4 后端推送给 UE 的 WebSocket 消息格式

```json
{
  "type": "telemetry",
  "drone_id": "d1",
  "x": 10000,       // UE 世界坐标偏移 X，厘米
  "y": 20000,       // UE 世界坐标偏移 Y，厘米
  "z": 5000,        // UE 世界坐标偏移 Z，厘米
  "yaw": 45.0,      // UE5 Yaw 角，度（已翻转）
  "pitch": 0.0,     // 俯仰角，度
  "roll": 0.0,      // 横滚角，度
  "speed": 3.5,     // 速度，m/s
  "battery": 85     // 电量，%
}
```

> 注意：`x/y/z` 是相对无人机上电锚点的**偏移量**，不是 UE5 世界绝对坐标。UE5 端需将其叠加到锚点世界坐标上，才能得到镜像机的最终世界坐标。

### 2.5 UE5 端最终定位计算

UE5 收到遥测数据后，镜像机世界坐标计算方式：

```
MirrorWorldLocation = AnchorWorldLocation + UE_offset

其中：
  AnchorWorldLocation = Cesium.TransformGPSToUnreal(anchor_lat, anchor_lon, anchor_alt)
  UE_offset = (x, y, z)  // 后端推送的偏移量（厘米）
```

**锚点建立流程**：
1. 后端收到无人机首包遥测时，记录当前 GPS 坐标作为锚点
2. 后端通过 WebSocket 推送 `power_on` 事件，**直接携带** `gps_lat/gps_lon/gps_alt`
3. UE5 收到事件后，调用 Cesium API 将 GPS 转换为 UE5 世界坐标，存为 `AnchorWorldLocation`
4. 后续每帧：`MirrorWorldLocation = AnchorWorldLocation + UE_offset`

> `GET /api/drones/{id}/anchor` 接口保留，仅用于 UE5 重启后补偿查询已上电无人机的锚点，正常流程不需要调用。

## 三、链路 B：UE5 → 后端 → 无人机（控制指令）

### 3.1 数据流概览

```
UE5 前端
  │  用户点击地图，生成目标点（UE5 世界坐标，厘米）
  │  减去本地存储的 AnchorWorldLocation，得到相对偏移
  │  通过 WebSocket 发送 move 指令（相对偏移，厘米）
  ↓
后端服务
  │  接收 WebSocket 控制指令（已是相对偏移，无需再减锚点）
  │  执行 UE 偏移 → NED 转换
  │  打包为带 command_id/sequence 的 UDP JSON 控制指令并重复发送
  ↓
PX4 无人机
     接收 UDP 控制包，执行飞行
```

### 3.2 UE5 端预处理（发送前）

UE5 在发送指令前，先将目标点从绝对世界坐标转换为相对锚点的偏移：

```
UE_offset_X = UE_target_X - AnchorWorldLocation_X
UE_offset_Y = UE_target_Y - AnchorWorldLocation_Y
UE_offset_Z = UE_target_Z - AnchorWorldLocation_Z
```

> `AnchorWorldLocation` 存储在 UE5 端，由锚定流程（见 2.5）建立。后端不持有此数据。

### 3.3 UE5 发送给后端的 WebSocket 消息格式

```json
{
  "mode": "move",
  "drone_id": "d1",
  "x": 1000,    // 相对锚点的 UE 偏移 X，厘米
  "y": 2000,    // 相对锚点的 UE 偏移 Y，厘米
  "z": 500      // 相对锚点的 UE 偏移 Z，厘米
}
```

> 目标点指令使用 `mode` 字段，取值为 `move` / `scout` / `patrol` / `attack`。当前阶段四种模式均按“移动到目标点”处理；后端统一转换为 `mode=move` 的 UDP JSON 控制指令。悬停通过 pause 触发 `mode=hold`。

### 3.4 后端坐标转换（UE 偏移 → NED）

```
NED_North = UE_offset_X × 0.01    // 厘米 → 米
NED_East  = UE_offset_Y × 0.01    // 厘米 → 米
NED_Down  = UE_offset_Z × (-0.01) // 取反，厘米 → 米
```

### 3.5 后端发送给机载电脑的 UDP JSON 控制指令

```json
{
  "protocol": "ue5_drone_control",
  "version": 1,
  "type": "control",
  "session_id": "backend-...",
  "command_id": "backend-...-d1-s123",
  "sequence": 123,
  "drone_id": 1,
  "slot": 1,
  "mode": "move",
  "issued_at_unix_s": 1783650000.0,
  "sent_at_unix_s": 1783650000.2,
  "target": {
    "frame": "NED",
    "reference": "power_on_origin",
    "unit": "m",
    "north": 10.0,
    "east": 20.0,
    "down": -5.0
  },
  "delivery": {
    "repeat_index": 1,
    "repeat_total": 5
  }
}
```

可靠性与安全规则：

1. 后端对同一逻辑 `move` 指令使用相同 `session_id + command_id + sequence` 重复发送 5 次。
2. Jetson 默认只在 2.5 秒窗口内收到至少 3 个 `repeat_index` 不同、执行负载完全相同的数据报后应用 setpoint。
3. Jetson 按 `sequence` 拒绝迟到旧指令；同一 `command_id` 若携带不同坐标则整组丢弃。
4. `slot` 必须与本机脚本启动参数一致，防止多机端口串包。
5. `target` 已经是 PX4 可直接使用的 NED 米坐标；Jetson 不再执行坐标转换。
6. 坐标原点固定为无人机本次上电时的 NED 原点，与 `vehicle_odometry.position` 的原点一致。
7. Jetson 确认应用后，在 YAML 遥测的 `control_ack` 字段返回最后执行的命令 ID 和序号。

## 三·五、链路 B 特殊情况：阵列任务派发的坐标转换

### 背景

阵列路径在**队列编辑关卡**中编辑并保存为 JSON，该关卡没有 Cesium，使用普通 UE 世界坐标系（原点为编辑器场景原点，单位：厘米）。

派发到预演关卡（CesiumWorld）时，路径坐标需要从"队列编辑关卡绝对坐标"转换为"相对 GPS 锚点的 UE 偏移"，才能被后端正确解析并转成 NED 发给 PX4。

### 参考机机制

派发时自动选取 **DroneId 最小的那架无人机**作为参考机。

参考机满足以下条件：
- 已收到 `power_on` 事件，建立了 `AnchorWorldLocation`（GPS→UE 世界坐标）
- 其路径在队列编辑关卡中的第一个航点坐标，作为整个阵列的**编辑坐标原点**

> 派发前若参考机尚未获得 GPS 锚点，UI 会阻止派发并提示等待上电信息。

### 坐标转换公式

```
设：
  RefEditOrigin   = 参考机路径第一个航点的队列编辑坐标（cm）
  RefAnchor       = 参考机的 AnchorWorldLocation（Cesium GPS→UE 世界坐标，cm）
  Waypoint_Edit   = 某航点的队列编辑坐标（cm）

则发给后端的 UE 偏移：
  Waypoint_UE_offset = (Waypoint_Edit - RefEditOrigin) + RefAnchor
```

> **2026-07-10 协议复核警告：** 上式实际得到的是 Cesium UE 世界坐标，不是“相对某架无人机上电锚点的偏移”，因此不能直接进入后端现有 `UeOffsetToNed`。手动匹配时应再减去目标无人机自己的 `DroneAnchorWorldLocation`；自动分配时因为发送请求前尚不知道路径分给哪架飞机，应由后端利用各机 GPS 上电锚点计算“参考机原点 → 目标机原点”的 NED 差，再转换为该机本地 NED 目标。当前 `SequenceDispatchPanelWidget` 的阵列重映射仍需按此方案单独改造。

### 含义解释

| 分量 | 含义 |
|------|------|
| `Waypoint_Edit - RefEditOrigin` | 航点相对参考机初始位置的形状偏移，保持队列几何形状不变 |
| `+ RefAnchor` | 将形状偏移叠加到参考机真实 GPS 位置上，映射到 Cesium 世界坐标 |

### 集结行为

后端收到阵列任务后，会先执行集结流程：

- **参考机**（DroneId 最小）：初始航点即自身当前位置（偏移为零），无需移动
- **其他无人机**：飞到各自初始航点对应的 GPS 位置（相对参考机的真实地理偏移）
- 全部就位后，后端广播 `assembly_complete`，所有无人机同步开始执行路径

### 数据流

```
队列编辑关卡（JSON）
  │  Waypoint.Location = 编辑坐标（普通 UE 世界坐标，cm）
  ↓
SequenceDispatchPanelWidget::OnDispatchClicked
  │  取参考机 AnchorWorldLocation（已由 power_on 事件建立）
  │  取参考机路径第一个航点作为 RefEditOrigin
  │  对每个航点：Wp.Location = (Wp.Location - RefEditOrigin) + RefAnchor
  ↓
DroneNetworkManager::SendArrayTaskFromData
  │  发给后端的坐标已是"相对 GPS 锚点的 UE 偏移（cm）"
  ↓
后端
  │  UE 偏移 → NED：X×0.01, Y×0.01, Z×(-0.01)
  │  执行集结 + 路径执行
  ↓
PX4 无人机
```

### 与单点移动指令的区别

| 指令类型 | 坐标来源 | 减去锚点的位置 |
|----------|----------|----------------|
| 单点 move | 用户点击 Cesium 地图，UE 世界坐标 | `SendMoveCommand` 调用前由 PlayerController 减去 `AnchorWorldLocation` |
| 阵列派发 | JSON 文件，队列编辑坐标 | `OnDispatchClicked` 中转换：`(Wp - RefEditOrigin) + RefAnchor` |

## 四、GPS 锚点与 Cesium 坐标转换（特殊说明）

### 4.1 为什么需要 GPS 锚点

PX4 的 odometry `position` 字段是相对上电位置的 NED 偏移，**不是地理绝对坐标**。要在 Cesium 地图上正确显示无人机位置，必须知道上电时的 GPS 坐标（锚点），才能将 NED 偏移映射到地球上的真实位置。

### 4.2 GPS → UE5 世界坐标（Cesium API）

此转换**只在 UE5 端执行**，后端不参与：

```cpp
// 正确调用方式（CesiumCoordinateService.cpp 实现）
FVector AnchorWorldLocation = Georeference->TransformLongitudeLatitudeHeightPositionToUnreal(
    FVector(Longitude, Latitude, Altitude)  // 注意参数顺序：X=经度, Y=纬度, Z=海拔
);
// 返回值：UE5 世界坐标（厘米）
// 前提：CesiumGeoreference Actor 必须位于 UE5 世界原点 (0,0,0)
```

**关键约束：CesiumGeoreference Actor 必须放在世界原点**

`TransformLongitudeLatitudeHeightPositionToUnreal` 返回的是相对于 CesiumGeoreference Actor 坐标系的局部坐标。当 Actor 位于世界原点时，局部坐标 = 世界坐标，转换结果正确。如果 Actor 不在原点，需要额外乘以 Actor Transform，但这会引入地球旋转矩阵导致坐标错误。**标准做法是始终将 CesiumGeoreference Actor 放在 (0,0,0)**。

**验证方法：将 Origin GPS 本身转换，应得到 (0,0,0)**

```
Georeference Origin = (39.981°N, 116.347°E, 43m)
TransformLongitudeLatitudeHeightPositionToUnreal(116.347, 39.981, 43) → (0, 0, 0) ✓

目标点 = (39.981°N + 0.001°, 116.347°E, 50m)（约100m北、7m高）
→ UE 世界坐标应接近 (-几百, 0, 700) 而非 (-2963171, -216732, -6210)
```

如果 Origin GPS 转出来不是 `(0,0,0)`，说明 CesiumGeoreference Actor 不在世界原点，或 Origin 设置和关卡地形不匹配。

### 4.3 本项目 CesiumWorld 关卡的轴向对齐情况

经测试（2026-05-23）：

```
North 方向在 UE5 中 = (0, -1, 0)   即北 → UE -Y 轴
East  方向在 UE5 中 = (1,  0, 0)   即东 → UE +X 轴
```

**这意味着本项目 Cesium 场景的地球旋转了 90 度**，北不是 +X 而是 -Y。

对 GPS→UE 锚点转换**没有影响**（Cesium API 内部处理旋转，结果始终正确）。

对后端的 NED→UE 线性偏移转换**有影响**：后端假设 North→X、East→Y，但实际 North→-Y、East→+X。这意味着后端推送的遥测偏移方向和实际地图方向不一致——**当前阶段可接受**（误差仅在偏移方向，锚点本身正确），但如需精确对齐，后端应按实际轴向调整转换公式，或在 Cesium 场景中修正地球朝向使 North 对齐 +X。

### 4.4 为什么不用简单线性公式替代 Cesium API

简单线性公式（`UE_X = NED_North × 100`）存在两个问题：

1. **轴向不对齐**：Cesium 默认在 Georeference 原点处 UE5 +X 对应东方向，不是北方向，直接套公式会导致 North/East 接反。
2. **球面误差**：线性公式假设地球是平的，离 Georeference 原点越远误差越大（几公里范围内误差显著）。

**正确做法**：
- 锚点 GPS → UE5 世界坐标：用 Cesium API（UE5 端）
- NED 偏移 → UE5 偏移：用线性公式（后端，近距离范围内可接受）
- 最终位置 = Cesium 锚点 + 线性偏移

## 五、转换公式汇总

### 链路 A（遥测，无人机 → UE5）

| 步骤 | 执行方 | 输入 | 输出 | 公式 |
|------|--------|------|------|------|
| NED → UE 偏移 | 后端 | NED (m) | UE offset (cm) | X×100, Y×100, Z×(-100) |
| 四元数重排 | 后端 | [w,x,y,z] | FQuat | (x, y, -z, w)，Yaw 取反 |
| GPS → UE 世界坐标 | UE5 | lat/lon/alt | UE world (cm) | Cesium API |
| 最终定位 | UE5 | 锚点 + 偏移 | 镜像机位置 | AnchorLoc + UE_offset |

### 链路 B（控制，UE5 → 无人机）

| 步骤 | 执行方 | 输入 | 输出 | 公式 |
|------|--------|------|------|------|
| UE 绝对 → UE 偏移 | **UE5** | UE world (cm) | UE offset (cm) | target - AnchorWorldLocation |
| UE 偏移 → NED | 后端 | UE offset (cm) | NED (m) | X×0.01, Y×0.01, Z×(-0.01) |
| JSON 发送 | 后端 | NED (m) | UDP JSON | `frame=NED, reference=power_on_origin, unit=m`，同一指令 5 次重发 |
| 阈值确认 | Jetson | UDP JSON | PX4 setpoint | 默认 2.5 秒内 3 个唯一重复包才应用，旧 sequence 丢弃 |

## 六、断联重连处理

无人机断联后重新上电，odometry 原点重置（新的上电位置），旧锚点失效。

**重连流程**：
1. 后端检测到重连，重新记录新的 GPS 锚点
2. 后端通过 WebSocket 推送 `reconnect` 事件，**直接携带**新的 `gps_lat/gps_lon/gps_alt`
3. UE5 收到事件，重新调用 Cesium API 计算新锚点，更新 `AnchorWorldLocation`
4. 镜像机位置从新锚点重新开始计算

## 七、常见错误排查

| 现象 | 可能原因 | 检查点 |
|------|----------|--------|
| 镜像机位置偏移方向错误 | NED 轴向接反 | 确认 X=North, Y=East, Z=Down |
| 镜像机 Z 轴方向相反 | Z 取反遗漏 | 检查 `NED_Down × (-100)` |
| 镜像机 Yaw 方向相反 | Yaw 翻转遗漏 | 检查 `Yaw_UE = -Yaw_NED` |
| 镜像机位置整体偏移 | 锚点未正确建立 | 检查 `power_on` 事件和 Cesium API 调用 |
| 重连后镜像机位置跳变 | 锚点未更新 | 检查 `reconnect` 事件处理逻辑 |
| 控制指令目标点偏移 | 未在 UE5 端减去锚点 | 检查 UE5 发送前是否执行 target - AnchorWorldLocation |
| 单位错误（米/厘米混淆） | 乘以系数错误 | UE5 内部用厘米，NED 用米，×100 或 ×0.01 |
| GPS 转出 UE 坐标距离原点极远（万米量级） | 测试用 GPS 和 Georeference Origin 相差太远 | 用和 Origin 经纬度接近的 GPS 坐标测试，Origin 转换应得 ~(0,0,0) |
| GPS 转出 UE 坐标为负 Z（地下） | 测试 GPS 海拔低于 Origin 海拔，或测试点经纬度偏差导致地形高度差 | 测试点经纬度应与 Origin 接近，海拔应高于 Origin 海拔 |

## 八、Cesium API 使用陷阱（排查记录 2026-05-23）

### 8.1 `TransformLongitudeLatitudeHeightPositionToUnreal` 的正确用法

```cpp
// ✓ 正确：CesiumGeoreference Actor 在世界原点时，直接返回 UE 世界坐标
FVector WorldPos = Georeference->TransformLongitudeLatitudeHeightPositionToUnreal(
    FVector(Longitude, Latitude, Altitude));

// ✗ 错误：再乘以 Actor Transform 会引入地球旋转矩阵，导致坐标偏差万米量级
FVector LocalPos = Georeference->TransformLongitudeLatitudeHeightPositionToUnreal(...);
FVector WorldPos = Georeference->GetActorTransform().TransformPosition(LocalPos); // 不要这样做
```

### 8.2 `GeoTransforms::TransformLongitudeLatitudeHeightToUnreal` 的 origin 参数

此低层 API 要求传入浮点原点的 ECEF 偏移，传 `(0,0,0)` 不等于"从 Georeference Origin 计算"，而是不减去任何偏移，结果是地球局部坐标系中的绝对坐标（离原点数万米）。**优先使用 `ACesiumGeoreference` 的高层 API，不要直接调用 `GeoTransforms`。**

### 8.3 `WorldToGeographic` 的正确用法

```cpp
// ✓ 正确：CesiumGeoreference Actor 在世界原点时直接传入世界坐标
FVector LLH = Georeference->TransformUnrealPositionToLongitudeLatitudeHeight(WorldLocation);

// ✗ 错误：先 InverseTransform 再转换（当 Actor 在原点时 InverseTransform 是恒等变换，
//   但逻辑上多余，且 Actor 不在原点时会出错）
FVector Local = Georeference->GetActorTransform().InverseTransformPosition(WorldLocation);
FVector LLH   = Georeference->TransformUnrealPositionToLongitudeLatitudeHeight(Local);
```

### 8.4 测试 GPS 坐标必须和 Georeference Origin 接近

mock server 默认 `gps_lat=39.9, gps_lon=116.3`，本项目关卡 Origin 可能是 `39.981, 116.347`。两者相差约 10km，转出来的 UE 坐标差距巨大，会导致无人机出现在远离地图的位置或地下。

**测试时务必将 mock server 的 GPS 坐标修改为和关卡 Origin 一致（或接近）的值。**
