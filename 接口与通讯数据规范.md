# 接口与通讯数据规范

> 版本：v2.1
>
> 日期：2026-07-10
>
> 适用代码：`Backend/` 与 `Drone/jetson_bridge.py`
> 本次变更：新增全机任务状态 `drone_task_state`，用于状态面板同步；UE-only 敌对目标发现和攻击确认暂不进入后端协议。

本文是当前接口和通讯数据格式的权威说明。坐标推导与 Cesium 细节见《坐标系转换说明.md》，模块边界与运行时关系见《架构设计.md》。

## 一、通讯总览

```text
UE5 前端
  │ HTTP REST：注册、查询、阵列任务
  │ WebSocket：控制请求、遥测/事件/告警
  ▼
DroneBackend（C++）
  │ UDP JSON v1：NED 目标；默认 5Hz、move 重发 5 次
  ▲ UDP YAML：PX4 遥测 + Jetson control_ack
  ▼
Jetson：Drone/jetson_bridge.py
  │ ROS2：50Hz OffboardControlMode + TrajectorySetpoint
  ▲ ROS2：VehicleOdometry / Status / GPS / Battery / CommandAck
  ▼
PX4 飞控
```

控制与遥测是两条独立 UDP 链路。后端 UDP 发送成功只说明数据报交给了操作系统；只有收到 Jetson 回传的 `control_ack`，才能确认目标已达到接收阈值并写入 Jetson 的 PX4 setpoint 缓存。

## 二、HTTP REST 接口

默认基址：`http://{backend_host}:8080`。响应内容类型为 JSON；错误响应统一包含：

```json
{"error":"错误说明"}
```

#### GET /

```json
{"status":"ok","http_port":8080,"ws_port":8081,"debug":true}
```

```json
    "gps_lon": 116.4074,
    "gps_alt": 50.0,
    "gps_fix": true,
    "armed": true,
    "offboard": true,
    "task_mode": "patrol",
    "task_state": "patrolling",

    "bit_index": 0,
    "mavlink_system_id": 1
  }

```

位置 `x/y/z` 是相对该机本次上电锚点的 UE 偏移，单位厘米。`gps_*` 是当前遥测中的WGS84位置。`task_*` 是后端保存的最新权威任务状态。`id` 是 UE 使用的数字 ID，`id_str` 是工具和协议兼容用的字符串 ID。

#### POST /api/drones

```json
{
  "name": "UAV1",
  "model": "PX4",
  "slot": 1,
  "ip": "192.168.30.104",
  "port": 8889,
  "video_url": "rtsp://example/live"
}
```

| 字段 | 要求 | 说明 |
|---|---|---|
| `name` | 推荐必填 | 唯一名称；缺省时使用自动 ID |
| `model` | 可选 | 型号 |
| `slot` / `slot_number` | 推荐必填 | 1～6，必须存在于 `port_map` 且不可重复 |
| `ip` | 可选 | 注册记录字段；当前并非控制数据报的最终目标地址 |
| `port` | 可选 | 注册记录字段；缺省时取 slot 的 `send_port` |
| `video_url` | 可选 | 视频流地址 |

成功响应（201）：

```json
{"id":1,"id_str":"d1","slot":1,"name":"UAV1"}
```

> 当前控制发送目标使用 `Backend/config.yaml` 的 `jetson.host` 和 `port_map[slot].send_port`。修改注册记录里的 `ip/port` 不会改变已经启动的后端→Jetson控制链路。

#### PUT /api/drones/{id}

可更新 `name`、`model`、`ip`、`port`、`video_url`；当前接口不更新 `slot`。

#### DELETE /api/drones/{id}

删除注册记录，并停止该机的 UDP 派发线程。

#### GET /api/drones/{id}/anchor

```json
{
  "drone_id": 1,
  "drone_id_str": "d1",
  "gps_lat": 39.9042,
  "gps_lon": 116.4074,
  "gps_alt": 50.0,
  "valid": true
}
```

锚点尚不可用时返回 HTTP 425。

### 2.3 阵列任务

#### POST /api/arrays

```json
{
  "array_id": "a1",
  "mode": "scout",
  "auto_assign": false,
  "paths": [
    {
      "pathId": 1,
      "drone_id": "d1",
      "bClosedLoop": false,
      "waypoints": [
        {
          "location": {"x": 0, "y": 0, "z": 300},
          "segmentSpeed": 2.0,
          "waitTime": 0.0
        }
      ]
    }
  ]
}
```

| 字段 | 说明 |
|---|---|
| `mode` | `scout`、`patrol` 或 `attack` |
| `auto_assign` | `true` 时后端用在线无人机自动匹配路径；否则每条路径必须给出 `drone_id` |
| `paths[].pathId` | 路径 ID；也兼容 `path_id` |
| `paths[].drone_id` | 无人机字符串 ID；也兼容 `droneId` |
| `paths[].bClosedLoop` | 是否闭环；也兼容 `closed_loop` |
| `waypoints[].location.x/y/z` | 约定为该执行机上电锚点下的 UE 相对偏移，单位厘米 |
| `segmentSpeed` | 航段速度，m/s |
| `waitTime` | 到点等待时间，秒 |

成功响应（201）：

```json
{"array_id":"a1","status":"assembling"}
```

> 当前已知边界：UE 的阵列重映射代码仍可能提交 Cesium/UE 世界坐标，而后端按“相对偏移”解释；多机上电 NED 原点又彼此不同。单机 `move` 链路已符合本规范，阵列任务在外场使用前必须完成每架机锚点补偿。详见《坐标系转换说明.md》中的阵列边界说明。

#### POST /api/arrays/{id}/stop

停止执行引擎和集结控制器：

```json
{"array_id":"a1","status":"stopped"}
```

## 三、WebSocket 消息协议

约定连接地址：`ws://{backend_host}:8081/ws`。服务端当前不按路径分流，但客户端统一使用 `/ws`。

### 3.1 UE5 → 后端

#### 移动类指令

```json
{
  "mode": "move",
  "drone_id": "d1",
  "x": 1000.0,
  "y": 2000.0,
  "z": 500.0,
  "request_id": "req-001"
}
```

`mode` 支持 `move`、`scout`、`patrol`、`attack`；当前实时入口都按目标点移动处理。`x/y/z` 是相对本机上电锚点的 UE 偏移厘米，后端转换为 NED `(10,20,-5)m`。

#### 暂停与恢复

```json
{"type":"pause","drone_ids":["d1","d2"],"request_id":"req-002"}
{"type":"resume","drone_ids":["d1","d2"],"request_id":"req-003"}
```

`pause` 暂停队列推进，并用最新实测 NED 位置生成持续 `hold`；如果尚无遥测，则使用后端已知的最后目标。`resume` 恢复队列。

#### command_ack

只有请求携带 `request_id` 时，成功接受后才返回：

```json
{
  "type":"command_ack",
  "command":"move",
  "mode":"move",
  "request_id":"req-001",
  "drone_id":1,
  "drone_id_str":"d1"
}
```

`command_ack` 只表示后端已接受并入队，不表示 Jetson 或 PX4 已执行。Jetson 的执行确认是 UDP YAML 中的 `control_ack`。

### 3.2 后端 → UE5

#### telemetry

```json
{
  "type":"telemetry",
  "drone_id":1,
  "drone_id_str":"d1",
  "x":100.0,
  "y":200.0,
  "z":300.0,
  "yaw":45.0,
  "pitch":0.0,
  "roll":0.0,
  "speed":3.5,
  "battery":85,
  "armed":true,
  "offboard":true,
  "gps_lat":39.9042,
  "gps_lon":116.4074,
  "gps_alt":50.0,
  "gps_fix":true
}
```

`x/y/z` 为 UE 相对偏移厘米；姿态角为度；`speed` 为 m/s；`battery=-1` 表示未知；`gps_*` 是当前飞控全球位置，不是控制使用的固定上电原点。推送频率随 Jetson 遥测频率，默认约 10Hz。

#### event

```json
{"type":"event","drone_id":1,"drone_id_str":"d1","event":"power_on","gps_lat":39.9,"gps_lon":116.3,"gps_alt":50.0}
{"type":"event","drone_id":1,"drone_id_str":"d1","event":"lost_connection"}
{"type":"event","drone_id":1,"drone_id_str":"d1","event":"reconnect","gps_lat":39.9,"gps_lon":116.3,"gps_alt":50.0}
```

`power_on` 和 `reconnect` 在有效 GPS 锚点存在时携带 GPS。重连意味着相对坐标原点可能已经变化，UE 必须重建锚点；后端同时清除与旧原点绑定的待执行指令。

#### 其他推送

| `type` | 关键字段 | 说明 |
|---|---|---|
| `alert` | `drone_id`、`alert`、`alert_type`、`value` | `low_battery` 或 `lost_connection` |
| `assembling` | `array_id`、`ready_count`、`total_count` | 集结进度 |
| `assembly_complete` | `array_id` | 集结完成并启动任务执行 |
| `assembly_timeout` | `array_id`、计数字段 | 集结超时 |
| `assignment_result` | `array_id`、`assignments[]` | `auto_assign=true` 的路径分配结果 |
| `drone_task_state` | 见下节 | 每架无人机的权威任务状态 |
| `error` | `code`、`message` | 请求错误 |

#### drone_task_state

```json
{
  "type": "drone_task_state",
  "drone_id": 1,
  "drone_id_str": "d1",
  "array_id": "a1",
  "mode": "patrol",
  "state": "patrolling",
  "current_wp": 2,
  "waypoint_count": 4,
  "detail": "",
  "updated_at": 1783650000.5
}
```

| `state` | 含义 |
|---|---|
| `standby` | 无活动任务或任务被停止 |
| `moving` | 单目标点移动 |
| `assembling` | 正在飞往阵列首航点 |
| `scouting` | 正在执行侦察路径 |
| `patrolling` | 正在执行巡逻路径 |
| `attacking` | 当前任务模式为攻击，正在飞向目标点 |
| `paused` | 操作员暂停 |
| `avoiding` | 预留的主动避障覆盖状态 |
| `completed` | 路径完成或攻击到点，保持悬停 |
| `error` | 任务错误；具体原因见 `detail` |

`current_wp` 从0开始。UE-only 的敌对目标发现、用户确认和本地攻击状态不通过本消息同步；UE面板需要单独叠加本地预演状态。UE状态面板应以本消息或 `GET /api/drones` 的 `task_*` 快照为准，不能用本地选择的命令模式代替后端真实状态。

## 四、后端 → Jetson：UDP JSON v1

### 4.1 数据报

每个 UDP 数据报是一个完整 UTF-8 JSON 对象：

```json
{
  "protocol":"ue5_drone_control",
  "version":1,
  "type":"control",
  "session_id":"backend-1783650000000000-ab12cd34",
  "command_id":"backend-1783650000000000-ab12cd34-d1-s1783650000000100",
  "sequence":1783650000000100,
  "drone_id":1,
  "slot":1,
  "mode":"move",
  "issued_at_unix_s":1783650000.0,
  "sent_at_unix_s":1783650000.2,
  "target":{
    "frame":"NED",
    "reference":"power_on_origin",
    "unit":"m",
    "north":10.0,
    "east":20.0,
    "down":-5.0
  },
  "delivery":{"repeat_index":1,"repeat_total":5}
}
```

| 字段 | 约束 |
|---|---|
| `protocol/version/type` | 必须是 `ue5_drone_control` / `1` / `control` |
| `session_id` | 后端进程会话 ID；后端重启后变化 |
| `command_id` | 同一逻辑指令的所有重复包保持一致 |
| `sequence` | 正整数、单调递增；Jetson 拒绝不大于最后已应用值的旧指令 |
| `drone_id` | 后端业务数字 ID |
| `slot` | 必须等于当前 Jetson 脚本启动 slot，防止端口串包 |
| `mode` | `move` 或 `hold` |
| `target` | 固定为本次上电原点下的 NED 米坐标 |
| `repeat_index` | 同一指令的包序号，从 1 开始；重复索引不重复计数 |
| `repeat_total` | `move` 默认为 5；持续 `hold` 为 0 |

### 4.2 接收确认策略

默认策略为 `3-of-5`：

1. 后端以 5Hz 对同一 `move` 发送 5 个包。
2. Jetson 在 2.5 秒窗口内收集同一 `command_id`。
3. 只有不同 `repeat_index` 的包计数；重复包不计数。
4. 同一 `command_id` 的 session、sequence、slot、mode、目标和坐标元数据必须完全一致；冲突时整组丢弃。
5. 收到 3 个有效包后，Jetson 才更新 setpoint，并记录 `control_ack`。
6. 新后端会话到来时清空旧待确认组；已应用 sequence 防止迟到旧包覆盖新目标。

该策略允许 5 包中丢失最多 2 包。它不等同于端到端必达：网络完全中断时不会执行，也不会伪造 ACK。

## 五、Jetson → 后端：UDP YAML 遥测

`Drone/jetson_bridge.py` 订阅 ROS2 PX4 输出话题，默认约 10Hz 发送：

```yaml
timestamp: 1713800000000000
position: [1.0, 2.0, -10.0]
q: [1.0, 0.0, 0.0, 0.0]
velocity: [0.5, 0.3, -0.1]
angular_velocity: [0.0, 0.0, 0.01]
arming_state: 2
nav_state: 14
local_position: [1.0, 2.0, -10.0]
local_velocity: [0.5, 0.3, -0.1]
gps_lat: 39.9042
gps_lon: 116.4074
gps_alt: 50.0
gps_fix: true
battery: 85
control_ack:
  session_id: backend-...
  command_id: backend-...-d1-s...
  sequence: 1783650000000100
  mode: move
  confirmed_packets: 3
  applied_at_unix_s: 1783650000.6
```

| 字段 | 坐标/单位 | 说明 |
|---|---|---|
| `timestamp` | 微秒 | PX4 时间戳 |
| `position` | NED，m | 相对本机上电原点；后端控制和状态判断使用此字段 |
| `q` | `[w,x,y,z]` | 机体姿态四元数 |
| `velocity` | NED，m/s | 速度向量 |
| `angular_velocity` | rad/s | 角速度 |
| `arming_state/nav_state` | PX4 枚举值 | 后端派生 `armed/offboard` |
| `gps_*` | WGS84/高度 m | 有效锚点和重连定位 |
| `battery` | 百分比 | 未知时为 -1 |
| `control_ack` | 应用层确认 | 仅在目标达到阈值且已写入 setpoint 后提供 |

当前后端解析 `control_ack` 的 session、command ID、sequence、mode 和确认包数；`applied_at_unix_s` 保留供日志和未来延迟统计。

## 六、端口和配置

### 6.1 固定端口映射

| slot | 后端遥测接收端口 | Jetson 控制接收端口 | 配置话题前缀 |
|---:|---:|---:|---|
| 1 | 8888 | 8889 | `/px4_1` |
| 2 | 8890 | 8891 | `/px4_2` |
| 3 | 8892 | 8893 | `/px4_3` |
| 4 | 8894 | 8895 | `/px4_4` |
| 5 | 8896 | 8897 | `/px4_5` |
| 6 | 8898 | 8899 | `/px4_6` |

Jetson 默认使用无前缀 PX4 话题；多机 ROS2 命名空间部署时，通过 `ROS_TOPIC_PREFIX=/px4_N` 显式覆盖。

### 6.2 关键参数

| 参数 | 默认值 | 所在端 | 说明 |
|---|---:|---|---|
| `heartbeat_hz` | 5 | 后端 | 后端到 Jetson 的 JSON 发送频率 |
| `command_repeat_count` | 5 | 后端 | 每条 move 的重发数，必须不小于 Jetson 确认阈值 |
| `lost_timeout_sec` | 10 | 后端 | 无遥测失联阈值 |
| `arrival_threshold_m` | 1.0 | 后端 | 航点到达阈值 |
| `COMMAND_CONFIRM_COUNT` | 3 | Jetson | 唯一重复包确认数 |
| `COMMAND_CONFIRM_WINDOW_SEC` | 2.5 | Jetson | 确认时间窗 |
| `MAX_ABS_TARGET_M` | 5000 | Jetson | 单轴目标绝对值安全上限 |
| `BACKEND_HOST` | `192.168.30.100` | Jetson | YAML 遥测发送目标 |
| `CONTROL_BIND_HOST` | `0.0.0.0` | Jetson | 控制 UDP 监听地址 |

`heartbeat_hz` 不是 PX4 Offboard 心跳。Jetson 始终独立以 50Hz 发布 ROS2 Offboard setpoint。

## 七、坐标与重连不变量

```text
UE 相对锚点偏移(cm) → 后端转换 → NED 相对上电原点(m) → Jetson原样写入 setpoint
PX4 NED相对位置(m) → Jetson原样上报 → 后端转换 → UE 相对锚点偏移(cm)
```

```text
N = Xue × 0.01
E = Yue × 0.01
D = -Zue × 0.01
```

- Jetson 不再对 JSON 目标做轴交换、符号翻转或单位换算。
- 遥测实际位置与控制期望目标必须分开保存；实际位置不能覆盖期望目标。
- `pause` 才使用最新实测位置创建 hold。
- 失联/重新上电可能改变 NED 原点，因此后端清空旧队列，UE 更新 GPS 锚点后才能继续下发相对目标。
- 每架机的 NED 原点独立，多机阵列必须把公共世界目标分别换算到各机原点。

## 八、联调判定与兼容性

推荐按下列日志顺序判断链路：

```text
后端 [UdpSender] / [Heartbeat]
  → Jetson [UDP-RX]
  → Jetson [COMMAND-PENDING]
  → Jetson [COMMAND-EXECUTE]
  → 后端 [ControlAck]
  → PX4 [PX4-ACK] / 状态变更
```

- 后端持续发送而 Jetson无 `[UDP-RX]`：检查 `jetson.host`、控制端口、路由、防火墙。
- Jetson 已 `[COMMAND-EXECUTE]` 而后端无 `[ControlAck]`：检查 YAML 回程、遥测端口和后端解析日志。
