# 多机地图交互与阵列控制架构改造清单

## 1. 文档目的

本文档基于 2026-04-01 的最新需求，重新梳理 `UE5DroneControl` 的多机地图交互与阵列控制架构改造方案。

本版不再沿用“先做地图、再补 5 个 FR”的拆法，而是围绕以下 6 项目标给出可行性判断和具体实现路径：

1. 增加“无人机阵列移动预制序列编辑”功能，并参考 RTS 控制方式
2. 真实无人机识别到目标后，可加速靠近目标点位
3. 保留“鼠标选中无人机后，再点击地图给该无人机派发命令”的当前 FR-02 能力
4. 鼠标中键选中无人机后弹出信息面板
5. 游戏运行时支持自由镜头与跟随镜头相互切换
6. 无人机传输 RTSP 视频流，并在 UE 端的信息面板中展示

---

## 2. 结论先行

### 2.1 总体可行性

这 6 项需求整体可行，但不能继续建立在“单一 32 字节掩码广播包 + TopDown 模板堆逻辑”的旧做法上。

当前工程已经有以下可复用基础：

- 已接入 `CesiumForUnreal`
- 已有 `DroneOps` 目录和基础类型、注册表、发送组件雏形
- 已有 `Variant_Strategy` 样板，可直接借鉴 RTS 式选择、框选和相机交互思路
- 已有 `ue_to_px4_bridge.py` 的 OFFBOARD 控制闭环

当前真正需要升级的，是 4 个关键点：

1. 控制协议
当前 32 字节协议只支持“一个共享目标点 + 多机掩码”，不支持真正的阵列步骤编辑，因为阵列步骤往往要求“每架无人机有各自不同的目标点”。

2. 执行层
当前 `ue_to_px4_bridge.py` 主要是“收到目标点就持续发位置 setpoint”，还没有“识别后加速逼近”的速度/加速度控制层。

3. 信息展示层
当前信息面板只适合挂遥测字段，还没有视频流接入链路。

4. 交互模式
当前 FR-02 的“点击选中，再点击地图派发”适合单机精确控制，但若要兼容 RTS 阵列编辑，必须引入明确的交互模式切换或命令上下文。

## 3. 可行性评估

| 需求 | 可行性 | 结论 | 主要原因 |
|------|------|------|------|
| RTS 式阵列序列编辑 | 高 | 推荐实施 | 项目已有 `Variant_Strategy` 可复用，但必须升级控制协议 |
| 真实无人机识别后加速靠近目标 | 中高 | 推荐实施 | `ue_to_px4_bridge.py` 已有 OFFBOARD 基础，需增加追踪/逼近状态机 |
| 保留 FR-02 单机点选派发 | 高 | 直接保留 | 当前已有基础能力，适合作为“精确控制模式” |
| 中键信息面板 | 高 | 直接实施 | `DroneOpsPlayerController` 和遥测组件雏形已存在 |
| 自由镜头 / 跟随镜头切换 | 高 | 直接实施 | `DroneOpsPlayerController` 已有切换入口，需补正式 Pawn |
| RTSP 视频在面板展示 | 中 | 可做，但需分阶段 | UE 端直接播放 RTSP 风险较高，建议增加本地视频中继或解码层 |

### 3.1 关于 RTSP 的专项判断

根据 Epic 官方 Media Framework 文档，UE 支持流媒体播放，并通过 `Stream Media Source` 接入受支持的流地址；官方教程明确提到流地址需要是“受支持的 URL 格式”，并提到不同播放器插件支持能力不同，示例中重点提到 HLS/MP4 等格式。官方文档中我没有找到对“RTSP 作为 Electra/通用方案的一线推荐格式”的明确说明，因此“UE 直接拉 RTSP 并稳定嵌入信息面板”属于高风险路径。

因此本项目推荐采用两阶段方案：

- 第一阶段：增加本地视频中继，把 RTSP 转成 UE 更稳妥的本地流地址，再在 UMG 中展示
- 第二阶段：如果后续实测对延迟要求更高，再考虑自定义低延迟解码路径

说明：
这是基于 Epic 官方文档的保守工程判断，不是说 RTSP 一定不可用，而是说不适合直接作为首选架构基线。

## 4. 现有架构的核心缺口

### 4.1 当前 32 字节广播协议不适合阵列编辑

当前 `DroneCommandSenderComponent` 的 32 字节结构为：

- `Timestamp`
- `X/Y/Z`
- `Mode`
- `DroneMask`
- `Sequence`

这意味着一条命令只能表达：

- “让掩码中的若干无人机一起去同一个目标点”

它无法表达：

- 同一步骤中 UAV1 去 A 点、UAV2 去 B 点、UAV3 去 C 点
- 阵列步骤里的速度、加速度、到达半径、停留时间
- 针对不同无人机的角色位和相对偏移

所以：

- FR-02 可以继续用当前协议
- RTS 式阵列预制序列不能继续用当前协议硬撑

### 4.2 当前桥接器没有“加速逼近”策略

当前 `ue_to_px4_bridge.py` 的控制逻辑本质上是：

- 持续发送 `OffboardControlMode(position=True)`
- 持续发送 `TrajectorySetpoint.position`
- `velocity` 和 `acceleration` 基本未用于主动逼近控制

这足以支撑“飞向某点”，但不够支撑：

- 识别到目标后进入追踪/逼近状态
- 远距离快速靠近，近距离减速收敛
- 在接近过程中限制最大速度、最大加速度和最小安全高度

### 4.3 当前信息面板没有视频链路

当前信息面板规划只覆盖：

- 遥测字段
- 在线状态
- 姿态、位置、高度等文字信息

还没有：

- 视频源配置
- 视频播放状态
- 视频失败重试
- 视频贴图或媒体纹理在 UMG 中的绑定链路

### 4.4 当前交互尚未区分“单机精确控制”和“RTS 阵列控制”

现在 FR-02 的交互是：

- 点击无人机
- 点击地图
- 派发目标点

这对单机很直观，但参考 RTS 的话，通常需要：

- 左键单选 / 框选
- `Shift` 追加选择
- 右键下达移动或编队命令
- 选择预制队形后，再指定阵列锚点或移动方向

如果不引入“交互模式”或“命令上下文”，后续会出现以下冲突：

- 左键到底是“选中”还是“派发目标点”
- 多选后点击地图是“群体移动到一点”还是“按阵列落位”
- 阵列编辑中是否还允许触发 FR-02 的单机点位派发

## 5. 推荐的目标架构

建议按以下 7 层组织：

1. 地图与空间底座层
负责 Cesium 地图、坐标转换、地图点击落点

2. 无人机注册与状态层
负责 `DroneId -> 遥测 / 选中 / 视频 / 锁控 / 当前任务`

3. 交互模式层
负责单机精确控制、RTS 多选、阵列编辑、信息查看四种上下文

4. 阵列编辑与调度层
负责预制序列、步骤编辑、相对偏移、播放/暂停/停止

5. 指令编排与分发层
负责把 UE 侧命令变成“单机指令”或“批量阵列指令”，再转给 Python 控制器

6. 真实无人机执行层
负责 OFFBOARD、追踪、逼近、加速、减速和最终收敛

7. 展示层
负责 HUD、信息面板、镜头模式、阵列状态、视频画面

---

## 6. 关键设计决策

## 6.1 保留 FR-02，但把它降级为“单机精确控制模式”

FR-02 仍然保留，而且仍然是非常重要的基础能力，但它不再承担阵列编辑的全部交互。

建议把运行时交互拆成两种主要模式：

### 模式 A：单机精确控制模式

- 左键点击无人机：主选中
- 再次点击地图：给当前主选中无人机派发目标点
- 用于精确指定某一架无人机到某个点

### 模式 B：RTS 阵列控制模式

- 左键单选
- 左键拖拽框选
- `Shift` 追加/取消多选
- 选择一个阵列预制序列
- 右键点击地图：指定阵列锚点或下一步的执行位置

这样可以同时满足：

- 保留你当前已经习惯的 FR-02 单机操作
- 新增符合 RTS 心智模型的多机控制方式

## 6.2 阵列序列编辑必须引入“批量多目标命令”

建议保留旧协议，并新增一条扩展命令链路：

### 控制协议 V1：保留现有 32 字节掩码命令

用途：

- 单机 FR-02
- 简单群体广播
- 悬停命令

特征：

- 高兼容
- 低改动
- 不能表达复杂阵列步骤

### 控制协议 V2：新增“批量多目标命令”

推荐形式：

- UE -> `multi_ue_controller.py` 发送扩展命令帧
- 扩展命令帧中包含多个无人机各自的目标点和运动参数
- `multi_ue_controller.py` 再逐架转发给各自的 `ue_to_px4_bridge.py`

推荐的数据结构：

```json
{
  "magic": "DXC2",
  "version": 2,
  "type": "batch_move",
  "sequence": 101,
  "issued_at": 1711267200.15,
  "commands": [
    {
      "drone_id": 1,
      "x": 12.5,
      "y": 3.0,
      "z": -5.0,
      "mode": "move",
      "cruise_speed": 4.0,
      "accel_limit": 1.5,
      "arrival_radius": 0.8,
      "hold_sec": 0.5
    },
    {
      "drone_id": 2,
      "x": 10.0,
      "y": 5.0,
      "z": -5.0,
      "mode": "move",
      "cruise_speed": 4.0,
      "accel_limit": 1.5,
      "arrival_radius": 0.8,
      "hold_sec": 0.5
    }
  ]
}
```

说明：

- 首版推荐在本机回环链路中使用可扩展 JSON 帧，优先保证调试便利和扩展性
- 旧版 32 字节 UDP 包继续保留，不需要推翻
- `multi_ue_controller.py` 需要同时支持 V1 和 V2

## 6.3 真实无人机逼近目标要引入“追踪/逼近状态机”

建议在 `ue_to_px4_bridge.py` 中增加 `ApproachController`，把执行过程拆成以下状态：

- `Idle`
- `MoveToWaypoint`
- `TargetAcquired`
- `AcceleratingApproach`
- `DeceleratingApproach`
- `HoldOnTarget`
- `Abort`

核心思路：

1. 普通派发命令时，仍按当前位置目标点飞行
2. 当接收到“目标已识别”事件后，切换到 `TargetAcquired`
3. 根据当前无人机位置、目标位置、目标速度估计和安全边界，生成逼近轨迹
4. 远距离阶段允许更高 `cruise_speed`
5. 接近目标时根据 `arrival_radius` 和 `decel_radius` 自动减速
6. 最终收敛到目标点附近后进入停留或悬停

在 PX4 Offboard 层面的具体建议：

- 保留 `TrajectorySetpoint.position`
- 增加对 `TrajectorySetpoint.velocity` 的前馈支持
- `OffboardControlMode` 从“仅 position”升级为“position + velocity feed-forward”模式
- 速度、加速度、安全高度和最大逼近速度全部配置化

说明：

- 这样做不需要推翻现有桥接器，只是在现有控制循环上增加更高级的执行策略
- 逼近策略建议放在 Python 桥接侧，而不是 UE 侧
- UE 侧只负责表达“期望命令”和“运动参数”，执行细节由桥接器收敛

## 6.4 RTSP 视频建议采用“中继优先，直拉保留实验入口”的架构

推荐架构如下：

```text
真实无人机 RTSP
  ↓
drone_video_gateway.py / FFmpeg / GStreamer 本地中继
  ↓
本机可播放流地址（优先 HLS / 本地可验证流）
  ↓
UE Media Player / Media Texture
  ↓
WBP_DroneInfoPanel 中的视频区域
```

推荐这样做的原因：

- UE 端直拉 RTSP 的兼容性与稳定性风险更高
- 本地中继层可以统一处理重连、转码、降分辨率、截图占位和视频健康检查
- 信息面板只关心“拿到一条可播放的本地视频地址”

建议分两步实施：

### 第一步：稳定展示优先

- 每架无人机配置 `rtsp_url`
- 启动本地视频中继
- 中继输出 `local_video_url`
- UE 用 `MediaPlayer + MediaTexture + UMG` 展示

### 第二步：低延迟优化

- 若 HLS 或转码延迟不可接受，再引入自定义低延迟解码方案
- 可选路线包括 FFmpeg 解码后共享纹理、共享内存、或自定义 C++ 视频组件

## 6.5 信息面板要升级为“遥测 + 视频”的统一容器

`WBP_DroneInfoPanel` 不再只是展示文字字段，而应分为两个区域：

- 左侧或上半区：遥测与状态信息
- 右侧或下半区：视频窗口

建议信息面板至少支持：

- 无人机名称 / ID
- 在线状态
- 当前任务状态
- 当前位置 / 高度 / 速度 / 姿态
- 目标状态：未识别 / 已识别 / 正在逼近 / 已到位
- 视频状态：未连接 / 连接中 / 播放中 / 重连中 / 失败
- 视频画面或占位图

## 7. 新的交互方案

## 7.1 运行时交互总览

建议采用以下默认键鼠语义：

| 输入 | 作用 |
|------|------|
| 左键点击无人机 | 单选无人机 |
| 左键拖拽 | 框选多架无人机 |
| `Shift + 左键` | 追加或取消选择 |
| 左键点击地图 | 在“单机精确控制模式”下给主选中无人机派发命令 |
| 右键点击地图 | 在“RTS 阵列模式”下设置阵列锚点或执行点 |
| 鼠标中键点击无人机 | 打开该无人机信息面板 |
| `F` | 自由镜头 / 跟随镜头切换 |
| `Q/E` | 自由镜头上下移动 |
| `W/A/S/D` | 自由镜头相对当前视角方向移动 |

## 7.2 为什么要保留“左键点地图派发”

你明确要求保留当前 FR-02，所以这里不建议粗暴改成“完全 RTS 右键派发”，而是做成双模式：

- 日常精确调度：继续用你熟悉的 FR-02 左键闭环
- 多机阵列控制：进入阵列模式后，再启用 RTS 式右键派发

这样对现有操作习惯破坏最小。

## 7.3 阵列编辑的推荐操作流

建议首版操作流如下：

1. 左键单选或框选多架无人机
2. 打开 `WBP_FormationEditor`
3. 选择一个预制序列
4. 在编辑面板中设置：
   - 阵列基准点类型
   - 成员相对偏移
   - 每一步持续时间
   - 是否启用加速逼近
5. 右键点击地图指定阵列锚点
6. 点击播放
7. 系统按步骤生成每架无人机的目标点并执行

## 12. 最终建议

这次需求变化后，项目不应该再被理解成“给现有点击移动再多补几个按钮”，而应该被理解成“从单机点位派发系统，升级为一个轻量的多机 RTS 式无人机调度台”。

最重要的架构判断有三个：

1. FR-02 要保留，但它只适合作为“单机精确控制模式”
2. 阵列预制序列必须新增 V2 批量多目标命令协议
3. RTSP 视频不要直接把风险压在 UE 播放器上，优先增加本地视频中继层

如果按这个方向推进，这套系统会同时具备：

- 单机精确操作
- 多机 RTS 式选择与阵列派发
- 真实无人机目标识别后的加速逼近
- 视频与遥测统一信息面板
- 自由镜头与跟随镜头切换

这也是当前工程里最稳、后续最容易扩展的一条路线。
