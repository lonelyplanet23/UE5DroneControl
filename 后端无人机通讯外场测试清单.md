# 后端与无人机通讯外场测试清单

> 版本：v3.1
> 目标：逐步验证每一条通讯链路，最终完成 UE 派发 → 后端接收 → 无人机执行 → UE 接收返回数据的完整闭环，并完成精度测量。
> 原则：每一步必须通过后再执行下一步。有真实无人机时用真机，无真机时用模拟器替代（见各节说明）。

---

## 一览：测试阶段划分

| 阶段 | 内容 | 工具 |
|------|------|------|
| **P0** | 环境准备与配置确认 | 手工检查 |
| **P1** | 后端基础验证（纯 Apifox，无需无人机） | Apifox |
| **P2** | **UE ↔ 后端基础连接验证（无需无人机）** | UE5 + Apifox |
| **P3** | 遥测入站验证（无人机 → 后端） | 真机 or simulate_telemetry.py + Apifox |
| **P4** | 控制出站验证（后端 → 无人机，纯后端调试接口） | Apifox + Wireshark |
| **P5** | 状态事件与异常链路验证 | Apifox + 真机操作 |
| **P6** | UE ↔ 后端完整通讯验证（含遥测、告警） | UE5 + Apifox |
| **P7** | 精度测量：1 米直飞测试（全链路） | UE5 + 真机 + 卷尺 |
| **P8** | 完整阵列任务执行（全链路闭环） | UE5 + 真机 + Apifox |

---

### P3-3 验证持续遥测推送

`power_on` 之后，Apifox WebSocket 窗口应持续收到（约 10Hz）：

```json
{
  "type": "telemetry",
  "drone_id": 1,
  "drone_id_str": "d1",
  "x": 0.0,
  "y": 0.0,
  "z": 0.0,
  "yaw": 0.0,
  "pitch": 0.0,
  "roll": 0.0,
  "speed": 0.0,
  "battery": 85
}
```

> **坐标说明（上电悬停时）：** 无人机静止在上电位置，NED position ≈ [0,0,0]，推送的 x/y/z 应接近 0。
>
> **UE 端处理逻辑（`OnWebSocketTelemetry`）：** 镜像机目标位置 = `AnchorWorldLocation` + `Snapshot.WorldLocation`（即锚点 + x/y/z 偏移）。

**坐标换算验证示例：**

若 Jetson 发来 `position: [1.0, 2.0, -3.0]`（NED 米），后端应推送：

```
x = 1.0 × 100 = 100.0  (North → UE X 方向，厘米)
y = 2.0 × 100 = 200.0  (East  → UE Y 方向，厘米)
z = -(-3.0) × 100 = 300.0  (Down 取反 → UE Z，厘米)
```

**通过标准：** 消息以约 10Hz 持续到达，x/y/z 换算与 NED 原始值对应。

**结果：** ☐ 通过  ☐ 失败  实测频率约：______Hz

## 附录 A：常见问题快速排查

| 现象 | 检查点 |
|------|--------|
| P1-1 失败，`GET /` 超时 | 后端是否已启动？防火墙是否放行 8080？ |
| P1-3 返回 409 | 执行 `DELETE /api/drones/1` 清理旧数据后重试 |
| P2-2 UE Log 出现 WS 连接失败 | 检查 WebSocketUrl 是否填写正确；防火墙是否放行 8081 |
| P2-3 UI 中无 UAV1 | 检查 BackendBaseUrl 是否正确；后端注册表是否有 UAV1 |
| P3-2 未收到 power_on | 确认 Jetson 发到的 IP 和端口（后端 IP:8888）正确；Wireshark 抓 UDP:8888 确认有包 |
| power_on GPS 坐标为 0 或异常 | 检查无人机 GPS 是否有信号（`gps_fix=true`）；室内无 GPS 信号时无法获得真实坐标 |
| P3-3 遥测 x/y/z 方向和预期相反 | 检查后端 NED→UE 转换：x=N×100，y=E×100，z=D×(-100) |
| P4-1 sent_count 不增长 | P3-2 是否已完成（必须先收到首包遥测才会启动心跳线程） |
| P4-3 last_ned_z 符号错误 | 确认后端 `z=UE_z×(-0.01)`，z 取反 |
| P6-2 UE 日志无 power_on | 检查 WebSocketUrl 是否指向后端正确 IP；后端日志确认 WS 客户端已连接 |
| P6-4 镜像机位置不对（漂移或跳变） | 确认 CesiumGeoreference Origin 已正确设置（DroneOpsGameMode.bUseCesiumCoordinates=true）；确认 power_on 事件 GPS 坐标已被 UE5 接收并转换为 AnchorWorldLocation |
| P6-5 Apifox WS 无 command_ack | 检查 UE5 是否已选中无人机（PrimarySelectedDroneId 有效）；检查 WsClient IsConnected 状态 |
| P6-5 command_ack 收到但后端 NED 坐标极大 | UE 发出的是绝对世界坐标，需确认 SendMoveCommand 调用处已减去 AnchorWorldLocation |
| P7-4 精度偏差大（>30cm） | 检查：①Jetson 时间戳是否漂移；②无人机 GPS 精度；③offboard 模式是否稳定保持 |
| P8-2 UE 点击执行无反应 | 检查阵列编辑中是否已为 UAV1 分配路径；HttpClient 是否就绪 |
| P8-3 集结长时间未完成 | 查看 `GET /api/debug/drone/1/state` 的 `position`，确认无人机确实在向集结点飞 |
| P8-5 镜像机不移动（无人机在飞） | 检查 WS 遥测包是否正常推送（Apifox WS 窗口确认）；检查 bHasGpsAnchor 是否已设置（需先收到 power_on） |

## 附录 B：Wireshark UDP 包解析方法

安装 Wireshark，抓取接口选择局域网网卡。

**过滤 Slot 1 控制包：**
```
udp.dst port 8889
```

**Python 解析 24 字节包：**
```python
import struct
data = bytes.fromhex("...")  # 从 Wireshark 复制 hex
ts, x, y, z, mode = struct.unpack('<dfffI', data)
print(f"ts={ts:.3f} x={x:.3f}m y={y:.3f}m z={z:.3f}m mode={mode}")
# mode=0: 悬停心跳  mode=1: 移动指令
```

## 附录 D：测试顺序依赖关系

```
P0（环境）
  └─→ P1（后端基础，纯 Apifox）
        └─→ P2（UE ↔ 后端基础连接，无需无人机）
              └─→ P3（遥测入站，需无人机/模拟器）
                    └─→ P4（控制出站，纯后端调试）
                          └─→ P5（异常链路验证）
                          └─→ P6（UE ↔ 后端完整通讯，含遥测/告警）
                                └─→ P7（精度测量，全链路）
                                      └─→ P8（阵列任务，全链路闭环）
```

每一层均依赖上一层完全通过。若某步失败，在该步骤完全修复并重新验证通过后，再继续下一步。
