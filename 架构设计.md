# 系统架构设计

> 版本：v1.0
>
> 日期：2026-07-10
>
> 适用范围：UE5 前端、C++ 后端、Jetson 机载桥接与 PX4 飞控

## 一、设计结论

当前“UE5 → 后端 → Jetson → PX4”的分层合理：UE5 负责交互和世界坐标，后端负责业务、状态和协议转换，Jetson 负责处理不可靠 UDP 并持续维持 PX4 Offboard，PX4 只接收本机 NED setpoint。

针对 UDP，单纯“收到任意若干包就执行”风险较高。当前实现将条件收紧为：同一后端会话、同一命令 ID、同一 sequence、相同执行负载、不同重复索引，并在时间窗内达到阈值才执行。这一设计在允许少量丢包的同时，可以防止重复包凑数、旧包回放、端口串包和同 ID 负载冲突。

需要明确的边界：`3-of-5` 是应用层抗丢包策略，不是严格可靠传输。对于未来需要“每条任务必达、明确失败、自动重试”的场景，应在现有 `control_ack` 上增加后端 ACK 超时状态机和有限重试；不建议把 PX4 的 50Hz Offboard 心跳依赖到局域网 UDP 上。

## 二、总体架构

```text
┌────────────────────────────────────────────────────────────────────┐
│ UE5 前端                                                           │
│ 阵列编辑 / 注册与状态 / 实时控制 / Cesium锚点 / 镜像机与影子机    │
│ HTTP Client + WebSocket Client                                     │
└────────────────────────────┬───────────────────────────────────────┘
                             │ HTTP :8080 / WebSocket :8081
┌────────────────────────────▼───────────────────────────────────────┐
│ DroneBackend（C++17）                                              │
│ HttpServer / WsManager / DroneManager / HeartbeatManager           │
│ UdpSender / UdpReceiver / CoordinateConverter / GPS Anchor         │
│ AssemblyController / ExecutionEngine                               │
└──────────────────────┬───────────────────────▲─────────────────────┘
                       │ UDP JSON 控制         │ UDP YAML 遥测+ACK
                       ▼                       │
┌────────────────────────────────────────────────────────────────────┐
│ Jetson：Drone/jetson_bridge.py                                     │
│ JSON校验 / 3-of-5 / 防旧包 / setpoint缓存 / 诊断日志 / YAML回传   │
└────────────────────────────┬───────────────────────────────────────┘
                             │ ROS2 50Hz setpoint / PX4输出话题
┌────────────────────────────▼───────────────────────────────────────┐
│ PX4 飞控：Offboard / VehicleOdometry / Status / GPS / Battery      │
└────────────────────────────────────────────────────────────────────┘
```

### 2.1 网络部署

| 节点 | 典型设备 | 主动连接/监听 |
|---|---|---|
| UE5 | Windows 显控机 | 主动连接后端 TCP 8080/8081 |
| DroneBackend | Windows 或独立后端机 | 监听 TCP 8080/8081、各 slot 遥测 UDP 端口 |
| Jetson | 机载电脑 | 监听控制 UDP 端口，向后端遥测 UDP 端口发送 |
| PX4 | 飞控 | 通过 MicroXRCEAgent 与 Jetson ROS2 通讯 |

后端监听 `0.0.0.0`。UE5 使用后端局域网 IP；后端使用 `config.yaml` 的 `jetson.host`；Jetson 使用 `BACKEND_HOST`。注册记录中的 `ip` 目前只用于持久化和展示，不是控制链路的权威路由配置。

## 三、组件职责

### 3.1 UE5 前端

| 模块 | 职责 | 不应承担 |
|---|---|---|
| 注册与状态 UI | 注册、编辑、轮询后端权威注册表、展示在线状态 | 直接管理 UDP 端口或 PX4 |
| Cesium/坐标服务 | GPS 锚点转 UE 世界坐标；世界坐标与相对锚点偏移换算 | 将世界坐标冒充相对坐标发送 |
| 实时控制 | 选择无人机、生成目标、发送 WebSocket move/pause/resume | 直接生成 NED setpoint |
| 阵列编辑与预演 | 航点编辑、模式配置、静态预演、碰撞提示 | 假设多架机共享同一个本地 NED 原点 |
| 镜像机 | 用后端遥测的相对偏移驱动真实状态显示 | 用期望目标替代实测位置 |
| 影子机 | 命令预览、插值和延迟可视化 | 作为真实无人机状态源 |

UE5 运行时状态中心为 `UDroneRegistrySubsystem`。后端注册表是网络侧权威数据源；UE 本地缓存保留历史条目和 UI 状态。`power_on/reconnect` 事件用于建立或更新每架机的 GPS 锚点。

### 3.2 C++ 后端

| 组件 | 核心职责 |
|---|---|
| `HttpServer` | REST 路由、WebSocket 请求解析、业务入口和事件推送 |
| `WsManager` | WebSocket 会话与广播 |
| `DroneManager` | 每机上下文、注册 ID/slot 映射、状态机、坐标转换、暂停/恢复 |
| `CommandQueue` | 每架机独立、线程安全的待执行目标 FIFO |
| `HeartbeatManager` | 按配置频率派发 JSON；move 有限重发；无 move 时持续 hold |
| `UdpSender` | 生成 session/command ID，序列化 UDP JSON v1 |
| `UdpReceiver` | 接收和解析 YAML 遥测及 `control_ack` |
| `CoordinateConverter` | UE 相对偏移厘米 ↔ NED 米 |
| `GpsAnchorManager` | 记录每架机有效 GPS 上电锚点 |
| `StateMachine` | Offline / Connecting / Online / Lost 与重连事件 |
| `AssemblyController` | 集结目标、到位判定、超时、自动分配入口 |
| `ExecutionEngine` | scout/patrol/attack 航点执行和基础避障 |

后端只把“相对锚点的 UE 偏移”转换为 NED。它不负责把 Cesium 世界坐标猜测为相对偏移，也不能假设不同无人机的本地 NED 原点一致。

### 3.3 Jetson 桥接

`Drone/jetson_bridge.py` 是后端与 PX4 之间的应用层协议终点，负责：

- 绑定对应 slot 的 UDP 控制端口。
- 严格校验协议版本、slot、sequence、坐标系、单位、数值范围和负载一致性。
- 在确认窗口内执行 `3-of-5` 去重计数。
- 只在确认后更新 `TrajectorySetpoint` 缓存。
- 独立以 50Hz 发布 `OffboardControlMode` 和最新 setpoint。
- 订阅 PX4 遥测，以 YAML 约 10Hz 回传后端。
- 将最后已应用目标作为 `control_ack` 附加到遥测。
- 输出能区分网络、协议、ROS2 与 PX4 拒绝位置的诊断日志。

Jetson 接收到的目标已经是 NED 米，不再次做轴变换、符号翻转或单位换算。

### 3.4 PX4

PX4 负责飞控闭环、安全检查、解锁/模式切换和实际轨迹跟踪。PX4 的命令 ACK 与 Jetson `control_ack` 含义不同：前者说明飞控接受或拒绝 ARM/OFFBOARD 等命令，后者说明应用层目标已经通过 UDP 确认并写入 setpoint 缓存。

## 四、数据模型与坐标不变量

### 4.1 三种位置必须分离

| 名称 | 含义 | 所在坐标 |
|---|---|---|
| 世界目标 | UE/Cesium 场景中的绝对显示位置 | UE 世界坐标，cm |
| 期望目标 | 本架机相对上电锚点的控制目标 | UE 相对偏移 cm / NED m |
| 实测位置 | PX4 当前报告的位置 | NED m，转换后为 UE 相对偏移 cm |

遥测实测位置不能覆盖期望目标，否则后端持续 hold 会把目标不断拉回当前点，表现为“命令收到但无人机不移动”。当前实现只在 `pause` 时用最新实测位置创建 hold。

### 4.2 坐标链路

```text
UE目标世界坐标 - 本机AnchorWorldLocation
  = UE相对锚点偏移(cm)
  → Backend: N=X×0.01, E=Y×0.01, D=-Z×0.01
  = NED相对本次上电原点(m)
  → Jetson原样写入 PX4 TrajectorySetpoint
```

遥测走完全相反的路径：

```text
UE_X=N×100, UE_Y=E×100, UE_Z=-D×100
```

每架机上电后拥有独立 NED 原点。公共世界目标必须针对每架机 GPS 锚点分别换算；同一个 NED 数值不代表多架机处于同一个地理点。

## 五、关键运行时流程

### 5.1 注册与上电

```text
UE5                Backend                 Jetson/PX4
 │ POST /api/drones   │                         │
 │───────────────────→│ 创建上下文、绑定slot    │
 │←───────────────────│ 返回数字id/id_str       │
 │                    │←──── YAML首个有效遥测 ──│
 │←─ WS power_on ─────│ 建立GPS锚点、启动派发    │
 │←─ WS telemetry ────│ NED→UE相对偏移           │
```

### 5.2 实时 move 与两级确认

```text
UE5                     Backend                    Jetson                PX4
 │ WS move + request_id   │                          │                    │
 │───────────────────────→│ 校验、UE cm→NED m、入队 │                    │
 │←─ command_ack ─────────│                          │                    │
 │                        │─ JSON repeat 1/5 ───────→│ pending 1/3        │
 │                        │─ JSON repeat 2/5 ───────→│ pending 2/3        │
 │                        │─ JSON repeat 3/5 ───────→│ apply setpoint      │
 │                        │                          │─ ROS2 50Hz ───────→│
 │                        │←─ YAML + control_ack ────│                    │
 │←─ telemetry ───────────│                          │←─ PX4 telemetry ───│
```

`command_ack` 是后端接收确认；`control_ack` 是 Jetson 应用确认。飞控是否真正进入 Offboard、是否接受解锁，以及是否到达目标仍要结合 PX4 状态和遥测判断。

### 5.3 move 完成重发后的 hold

后端完成 5 次 move 发送后，不立即消费和回放旧目标，而是生成新的 hold sequence，并持续发送同一目标。Jetson 对该 hold 首次仍执行阈值确认，之后用缓存目标维持 50Hz setpoint。新 move 或显式 pause 会替换当前 hold。

### 5.4 暂停与恢复

- `pause`：暂停队列，读取最新实测 NED，生成持续 hold。
- `resume`：恢复队列消费；不会凭空恢复已经因断联清除的旧目标。

### 5.5 失联与重连

```text
10s无遥测
  → Backend状态=lost
  → 停止该机派发线程
  → 清空旧相对坐标指令和最后目标
  → WS lost_connection + alert

新遥测到达
  → 记录新GPS锚点
  → 再次清除旧原点相关状态
  → 重启派发线程
  → WS reconnect
  → UE重建AnchorWorldLocation后再发送目标
```

### 5.6 阵列任务

```text
POST /api/arrays
  → 可选 Hungarian 自动匹配在线无人机与路径首点
  → AssemblyController 派发集结目标
  → 遥测位置与 arrival_threshold_m 比较
  → assembling 进度
  → 全部到位：assembly_complete → ExecutionEngine
  → 超时：assembly_timeout
```

当前阵列流程的算法链路存在，但多机坐标前提尚未完全闭合：UE 阵列重映射可能提供世界坐标，后端却按相对偏移转换；自动分配也需要把公共地理目标分别换算到每架机上电原点。外场多机任务必须先完成该补偿和专项测试。

## 六、并发与状态管理

| 执行单元 | 说明 |
|---|---|
| HTTP/WS 服务器线程 | 接收 UE 请求并推送状态 |
| Asio worker | 处理各遥测 UDP socket 的异步接收 |
| 每机 Heartbeat 线程 | 独立获取该机队列并发送 JSON |
| 主线程 | 每秒检查失联和集结超时 |
| Debug CLI 线程 | 仅 debug 配置下启动 |

共享注册表、无人机上下文、UDP 目标和回调均使用互斥保护；指令队列自身线程安全。Heartbeat 取队列时不再反向申请 `DroneManager` 总锁，避免断联停止线程时产生 join/锁互相等待。

每架机上下文至少包含：身份与 slot、连接状态机、命令队列、最新实测遥测、期望/hold 目标、GPS 锚点和最后 `control_ack`。这些字段不能合并为一份“当前位置”。

## 七、UDP 可靠性状态

### 7.1 后端状态

- 进程启动生成一次 `session_id`。
- 每条逻辑指令生成单调递增 `sequence`，由其组成 `command_id`。
- move 使用相同 ID 和负载发送 `repeat_index=1..5`。
- 发送统计记录成功数、失败数、当前 sequence、模式和重复进度。

### 7.2 Jetson 状态

- 以 `command_id` 维护待确认集合和执行负载指纹。
- 窗口内只累计不同 `repeat_index`。
- 新 session 清理旧待确认状态。
- `sequence <= highest_applied_sequence` 直接判旧。
- 同 ID 不同负载判冲突并丢弃整组。
- 确认后缓存目标并在 YAML 回传 ACK。

### 7.3 失败语义

| 现象 | 含义 |
|---|---|
| 后端 send 成功，Jetson无日志 | 数据报可能在路由、防火墙或端口处丢失 |
| Jetson pending 未达 3 | 丢包超过容忍范围或重复索引错误，不执行 |
| Jetson execute，后端无 ACK | 控制下行成功、遥测回程异常 |
| Jetson execute，PX4不动 | ROS2/MicroXRCE、模式、解锁或预检问题 |
| 后端收到 ACK，位置不变 | setpoint 已应用，但飞控未跟踪；看 PX4 状态/ACK |

## 八、可观测性设计

### 8.1 后端

- `[UdpSender]`：会话、目标地址、字节数和 NED。
- `[Heartbeat]`：指令开始、sequence、重发进度、hold 切换。
- `[UdpReceiver]`：遥测解析和来源 slot。
- `[ControlAck]`：Jetson 已应用的 command ID、sequence 和确认数。
- Debug HTTP：状态、队列、心跳、ACK 和模拟遥测。

### 8.2 Jetson

- `[UDP-RX]`：有效控制数据报。
- `[COMMAND-PENDING]`：确认进度与重复索引。
- `[COMMAND-EXECUTE]`：setpoint 已更新。
- `[COMMAND-STALE]` / `[COMMAND-CONFLICT]`：旧包或负载冲突。
- `[ONE-WAY-LINK]`：遥测正常发送但长期无有效控制。
- `[ROS-LINK]`：PX4 输入发布者没有订阅者。
- `[PX4-ACK]` / `[PX4-STATE]`：飞控命令和状态结果。

## 九、安全与演进建议

当前版本应继续遵循：

1. 后端和 Jetson 协议版本成对部署，禁止混用旧 24 字节二进制包。
2. `command_repeat_count >= COMMAND_CONFIRM_COUNT`。
3. 每次外场测试前核对 IP、slot、双向端口、ROS 前缀和 system ID。
4. 先在未装桨/受控环境验证 ARM/OFFBOARD，再进行运动测试。
5. 阵列多机功能在坐标补偿闭合前不得按“已验证”处理。

建议的后续增强按优先级排序：

1. 后端增加 `control_ack` 超时、有限重试和失败事件，形成明确的命令生命周期。
2. ACK 中增加 Jetson/PX4 状态、拒绝原因和实际应用延迟，并推送给 UE。
3. 为控制消息加入 HMAC 或迁移到带认证的可靠通道，防止局域网伪造控制包。
4. 将阵列航点契约升级为显式 `frame`、`reference`、`unit`，并为每架机做 GPS/NED 原点补偿。
5. 增加真实 UDP 丢包、乱序、重复、会话切换和进程重启的端到端测试。
